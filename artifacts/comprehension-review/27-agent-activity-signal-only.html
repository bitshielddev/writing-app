Agent activity becomes signal-only

This change removes raw diagnostic payloads from the agent activity channel, reduces the set of activity kinds, and makes renderer activity updates asynchronous in small batches. The activity UI now shows curated status, titles, and text only.

Review target: current working tree diff

Nearby history checked: d9d9eba, 567d2f1, d34e234, 0cf94c8, a937f24

Relevant changed files: 14, centered on contracts, agent event conversion, diagnostics, and renderer activity state.

Boundary tightened

AgentActivity no longer accepts payload, reasoning, or provider. Strict contract validation now rejects those fields and variants.

Producers emit fewer rows

Streaming message updates and tool start/update events no longer become activity rows. Final message/tool events still produce user-visible milestones.

Renderer batches activity

Desktop activity events are queued and flushed by animation frame or timer fallback, so consumers must expect a short delay before state changes.

The boundary changed first

The central behavior is visible in src/contracts/events.ts. Activity has become a narrow, display-oriented object: an id, kind, timestamps, title, optional text, and optional status. There is no generic JSON payload slot.

export const AgentActivityKindSchema = Type.Union([
  Type.Literal("lifecycle"), Type.Literal("message"), Type.Literal("tool"),
  Type.Literal("loop"), Type.Literal("error"),
]);
const activityProperties = {
  id: identifier,
  kind: AgentActivityKindSchema,
  timestamp,
  title: text(500),
  text: Type.Optional(text(100_000)),
  status: Type.Optional(AgentStatusSchema),
};
export const AgentActivityInputSchema = Type.Object(activityProperties, strict);
export const AgentActivitySchema = Type.Object(
  { ...activityProperties, updatedAt: timestamp },
  strict,
);

The test added in src/contracts/contracts.test.ts makes the contract intent explicit. The valid activity passes, while extra raw fields and the removed variants fail:

Maintenance invariant

Any new activity field must be added to the contract before it can cross the agent process, main process, or renderer boundary. Because these schemas use strict, adding a field only at a producer will be rejected rather than ignored.

The old safety valve is gone, by design of this diff

Previously, arbitrary payloads were allowed but passed through safeActivityPayload, which recursively redacted sensitive keys and capped serialized size. This diff deletes both src/domain/activity/payload.ts and payload.test.ts. The code no longer tries to make raw payloads safe; it prevents them from being activity data.

Before

add(input: Omit<AgentActivity, "updatedAt" | "payload"> & { payload?: unknown }) {
  const activity: AgentActivity = {
    ...input,
    updatedAt: Date.now(),
    ...(input.payload === undefined ? {} : { payload: safeActivityPayload(input.payload) }),
  };
}

Now

add(input: Omit<AgentActivity, "updatedAt">) {
  const activity: AgentActivity = {
    ...input,
    updatedAt: Date.now(),
  };
}

The same simplification happens in src/utility/agent/index.ts: postActivity now sends the supplied activity through the outgoing contract as-is. The guardrail has moved from redaction logic to schema shape plus producer discipline.

Risk to watch

If a future diagnostic needs raw provider data, the activity channel is now the wrong place for it. Reintroducing a generic payload field would bypass the main premise of this change unless it also recreates a careful redaction and size policy.

function observeMessage(event: MessageEvent): ActivityInput[] {
  if (event.type !== "message_end") return [];
  const parts = messageParts(event.message);
  const key = `${parts.role}:${parts.timestamp}`;
  const activities: ActivityInput[] = [];
  if (parts.text) {
    activities.push({
      id: `message:${key}`,
      kind: "message",
      timestamp: parts.timestamp,
      title: `${parts.role} message`,
      text: parts.text,
    });
  }
  if (parts.error) {
    activities.push({
      id: `error:${key}`,
      kind: "error",
      timestamp: parts.timestamp,
      title: "Provider error",
      text: parts.error,
      status: "error",
    });
  }
  return activities;
}

Two details matter for debugging later: message_update returns no activity, and thinking parts are no longer read by messageParts. If a developer expects model reasoning to appear in the UI, the current code path does not support that.

function observeToolActivity(event: ToolEvent, now: number): ActivityInput[] {
  if (event.type !== "tool_execution_end") return [];
  const failed = Boolean((event as { isError?: boolean }).isError);
  return [
    {
      id: `tool:${event.toolCallId}`,
      kind: "tool",
      timestamp: now,
      title: `${event.toolName} ${failed ? "failed" : "completed"}`,
      ...(failed ? { status: "error" as const } : {}),
    },
  ];
}

For tools, only the final event is visible. Failed tools get status: "error", but the raw result is deliberately not copied into text or payload.

What disappeared from the UI

AgentActivityList used to render a collapsible Raw payload block whenever item.payload existed. That block is removed. The visible activity row is now limited to kind, time, title, and optional text.

{item.text ? (

    {item.text}
  </p>
) : null}

This matches the contract change: the renderer no longer has a payload field to inspect. Any debugging workflow that relied on expanding raw provider requests or tool results in the Activity tab will need another source of diagnostics.

The renderer no longer updates activity synchronously

The activity feed still uses the same upsert semantics, including the 500-item cap, but live desktop events now pass through a queue. This reduces render churn when the agent emits bursts of milestones.

function upsertMany(items: AgentActivity[], activities: AgentActivity[]) {
  return activities.reduce(upsert, items);
}

const activityQueueRef = useRef<AgentActivity[]>([]);
const activityFrameRef = useRef<number | undefined>(undefined);
const activityFallbackRef = useRef<ReturnType<typeof setTimeout> | undefined>(undefined);
const scheduleActivityFlush = useCallback(() => {
  if (
    activityFrameRef.current !== undefined ||
    activityFallbackRef.current !== undefined
  ) return;
  if (
    typeof window !== "undefined" &&
    typeof window.requestAnimationFrame === "function"
  ) {
    activityFrameRef.current = window.requestAnimationFrame(flushActivity);
    activityFallbackRef.current = setTimeout(flushActivity, 100);
    return;
  }
  activityFallbackRef.current = setTimeout(flushActivity, 0);
}, [flushActivity]);

There are two timing paths. In a browser-like environment, the preferred path is requestAnimationFrame, with a 100ms timeout as a backup. Without animation frames, a zero-delay timeout is used. initialize and unmount both clear the queue and cancel any pending flush, so stale live activity should not land after a new hydration or after the hook is disposed.

const onDesktopEvent = useCallback((event: DesktopEvent) => {
  if (event.type === "agent.runtime") setRuntime(event.runtime);
  if (event.type === "agent.activity") {
    activityQueueRef.current.push(event.activity);
    scheduleActivityFlush();
  }
}, [scheduleActivityFlush]);

Testing and maintenance implication

Code that calls onDesktopEvent and immediately reads activity will now see the old state. The updated tests use waitFor or timer advancement for this reason.

Future maintainer notes

  • Do not add payload to an activity object and expect it to be ignored. The strict schema should reject it at the process contract.
  • If activity appears late in a test or UI trace, inspect the queue and flush path in useAgentController before looking for a lost desktop event.
  • If a tool failure needs human-readable details, add safe, bounded text at the producer. Do not copy raw tool results into activity.
  • Duplicate activity ids still update existing rows. In a batch, later items with the same id will win through activities.reduce(upsert, items).
  • Hydration through initialize is still immediate and trims history to AGENT_ACTIVITY_LIMIT. Live events are the async path.

Comprehension check

1. A Pi message_update event contains text and an error. What activity rows are produced?

None. observeMessage returns early unless event.type === "message_end". The test named "ignores streaming message updates" asserts this.

2. Why was safeActivityPayload removed instead of kept as a final sanitizer?

The changed contract no longer has a payload field. The new safety model is to keep raw/debug data out of agent activity entirely, then use strict schemas to reject extra fields.

3. What would likely break if a test expected result.current.activity to change immediately after onDesktopEvent?

The assertion could read stale state. Live activity is queued and flushed on an animation frame or timer, so tests need waitFor, timer advancement, or another async boundary.

4. A tool execution ends with isError: true and a large raw result. Which parts reach the activity UI?

The row has kind tool, a title like read failed, and status: "error". The raw result is not included as payload or text.

5. Where would you start debugging if provider request details no longer appear under "Raw payload"?

Start with the intentional removals: provider request/response listeners were deleted from createScribeExtension, provider is no longer an activity kind, and AgentActivityList no longer renders raw payload details.
