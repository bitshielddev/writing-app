Agent activity now reads in reverse chronological order

The staged change is small and targeted: the Activity tab in SuggestionDock no longer trusts the order of the incoming activity array. It displays the newest activity first, and a component test now locks that behaviour down.

The display order is now computed inside ActivityView

Previously, ActivityView mapped over items directly. The new code derives orderedItems first:

function ActivityView({
  items,
  runtime,
}: {
  items: AgentActivity[];
  runtime?: AgentRuntime;
}) {
  const orderedItems = [...items].sort(
    (left, right) => right.timestamp - left.timestamp,
  );

  return (
    ...
    <ol className="mt-5 grid gap-3" aria-label="Agent activity log">
      {orderedItems.map((item) => (
        <li key={item.id} ...>
          ...
        </li>
      ))}
      {!items.length ? (
        <li ...>Activity from this app launch will appear here.</li>
      ) : null}
    </ol>
  );
}

Line to notice: [...items] matters. JavaScript sort mutates the array it is called on. Cloning first keeps the React prop unchanged, so ActivityView does not reorder state owned by App.

The comparator sorts by timestamp descending, so a larger timestamp renders earlier. The empty-state condition still checks items.length, which is equivalent for this purpose because orderedItems is a copy of the same entries.

Follow one activity event to the screen

The staged diff does not change event storage, but the sorting decision is easier to reason about with the surrounding data flow in view.

New activity is appended. Existing activity is replaced in place. That means the state array is a storage/update structure, not a guaranteed presentation order. The new ActivityView makes the presentation rule explicit at render time.

// desktop/activity.ts

The activity ring preserves the original timestamp when an item with the same id is updated. Combined with the staged UI change, an updated item keeps its chronological position by creation time. If future product behaviour should instead move recently updated items to the top, the sort key would need to become updatedAt or use a two-key policy.

The new test captures the behaviour at the UI boundary

The test does not call ActivityView directly. It renders the dock, switches to the Activity tab through the same accessible button a user would use, then reads the visible activity card headings.

it("orders agent activity from newest to oldest", async () => {
  renderDock({
    activity: [
      { id: "middle", kind: "message", timestamp: 200, updatedAt: 200, title: "Middle message" },
      { id: "oldest", kind: "message", timestamp: 100, updatedAt: 100, title: "Oldest message" },
      { id: "newest", kind: "message", timestamp: 300, updatedAt: 300, title: "Newest message" },
    ],
  });

  await userEvent.click(screen.getByRole("button", { name: "Activity" }));

  expect(
    screen.getAllByRole("heading", { level: 3 }).map((heading) => heading.textContent),
  ).toEqual(["Newest message", "Middle message", "Oldest message"]);
});

Maintenance notes for this small sort

  • The UI sorts by timestamp, not by updatedAt. That matches the staged test, but it is a product choice. Updated streaming activity will not jump to the top if its original timestamp is older.
  • There is no explicit tie-breaker for equal timestamps. If several events share the same millisecond timestamp, their relative order is not specified by this component's comparator.
  • The test uses all level-3 headings after opening Activity. If future markup adds another h3 inside the Activity view, the test may need to scope the query to aria-label="Agent activity log".
  • The clone before sort is an invariant worth keeping. Removing it would mutate the incoming prop array during render.

Debugging prompt: if a user reports that the Activity tab "keeps moving old items," first check whether the unexpected movement is based on timestamp data from the agent event. If the report is about updated items not moving to the top, this implementation currently explains that behaviour.

Comprehension check

1. Why does the component use [...items].sort(...) instead of items.sort(...)?

Because sort mutates its receiver. items is a React prop owned by the parent, so cloning first lets the view compute presentation order without mutating parent state during render.

2. An activity item has timestamp: 100 and later receives an update with a newer updatedAt. Does this staged change move it above an item with timestamp: 300?

No. The comparator only reads timestamp. The surrounding activity ring also preserves the original timestamp when replacing an existing id.

3. What user path does the test exercise before checking order?

It renders the full SuggestionDock, clicks the accessible "Activity" button, then reads the rendered activity headings. That verifies the behaviour at the tab boundary rather than only testing a helper function.

4. What would be the likely symptom if the clone were removed and items.sort were used directly?

The Activity tab would still appear sorted, but the prop array passed from the parent could be mutated during render. That can make parent-owned state order harder to reason about and can create surprising effects if other code relies on the original array order.
