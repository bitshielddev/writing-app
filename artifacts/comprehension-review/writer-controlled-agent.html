Comprehension review

The Pi agent now waits for the writer

This staged change turns Scribe's autonomous Pi loop from launch-driven work into an explicit Start/Stop lifecycle. The Pi session still initializes, revisions still accumulate, and suggestions still persist through the same tools. The new boundary is whether the loop is enabled for this app launch.

Runtime

Default state is stopped

The loop constructor now sets status: "stopped", ignores restored active work, and keeps enablement in memory only.

Control path

Renderer asks Electron to start or stop

SuggestionDock exposes a persistent Start/Stop button. App calls the desktop bridge, which reaches the agent utility process through IPC/RPC.

Safety

Stop cancels active model work

The agent posts stopped, records lifecycle activity, and aborts an in-flight Pi stream. Later drain logic refuses to continue when disabled.

Start fetches a fresh storage seed before contacting the agent. That matters because the agent can be stopped while documents and sources continue to change.

The loop state now has two layers

ScribeLoopState now separates "what revision exists?" from "may the agent work right now?" The revision fields still track latest durable work. The new enabled flag gates cycles, yielding, failure state, and drain scheduling. 

// desktop/scribe-loop.ts
private enabled = false;
private cycleInProgress = false;

constructor(restored?: Partial<PersistedScribeLoopState>) {
  this.state = {
    latestRevision: restored?.latestRevision ?? -1,
    latestDocumentRevision: restored?.latestDocumentRevision ?? -1,
    activeRevision: undefined,
    activeDocumentRevision: undefined,
    yieldedRevision: restored?.yieldedRevision,
    cycleCount: restored?.cycleCount ?? 0,
    status: "stopped",
  };
}

beginCycle() {
  if (!this.enabled) return undefined;
  if (this.state.latestRevision < 0) return undefined;
  ...
}

Revision coalescing still happens while stopped. The difference is the status assigned when new work arrives. 

// desktop/scribe-loop.ts
revision(projectRevision: number, documentRevision: number) {
  if (projectRevision < this.state.latestRevision) return false;
  if (projectRevision === this.state.latestRevision) {
    return this.state.status === "working";
  }
  const isNewWork = projectRevision > (this.state.yieldedRevision ?? -1);
  this.state.latestRevision = projectRevision;
  this.state.latestDocumentRevision = documentRevision;
  if (isNewWork) {
    this.state.cycleCount = 0;
    this.state.error = undefined;
    this.yieldedInCycle = false;
    this.state.status = this.enabled ? "working" : "stopped";
  }
  return isNewWork;
}

Maintenance note. A stopped loop can still have unreviewed latest revisions. Do not treat stopped as "no work exists". It means work is not allowed to start until the writer enables it.

Start, stop, and abort semantics

Starting resets transient active-cycle fields and chooses working only if the latest revision is newer than the yielded revision. Stopping clears transient cycle state and rolls back the cycle count if a cycle had already begun. 

// desktop/scribe-loop.ts
start() {
  if (this.enabled) return false;
  this.enabled = true;
  this.cycleInProgress = false;
  this.state.activeRevision = undefined;
  this.state.activeDocumentRevision = undefined;
  this.state.cycleCount = 0;
  this.state.error = undefined;
  this.state.status =
    this.state.latestRevision > (this.state.yieldedRevision ?? -1)
      ? "working"
      : "waiting";
  return true;
}

stop() {
  if (!this.enabled) return false;
  this.enabled = false;
  if (this.cycleInProgress) {
    this.state.cycleCount = Math.max(0, this.state.cycleCount - 1);
  }
  this.cycleInProgress = false;
  this.yieldedInCycle = false;
  this.state.activeRevision = undefined;
  this.state.activeDocumentRevision = undefined;
  this.state.error = undefined;
  this.state.status = "stopped";
  return true;
}

The agent process wraps this state machine with user-visible activity and Pi cancellation. 

// desktop/agent.ts
async function stopAgent() {
  if (!configured || !session) {
    throw new Error("The agent is unavailable because Pi is not configured");
  }
  if (host.loop.stop()) {
    const timestamp = Date.now();
    host.persist();
    postRuntime(runtime());
    postActivity({
      id: `control:stopped:${timestamp}`,
      kind: "lifecycle",
      timestamp,
      title: "Agent stopped by writer",
      status: "stopped",
    });
    if (session.isStreaming) await session.abort();
  }
  return runtime();
}

Risk to keep in mind. Stop posts the stopped runtime before awaiting session.abort(). That matches the docs' claim of immediate cancellation from the UI point of view. If abort later fails or is slow, this code does not currently surface that as an error after the stopped state has been posted.

Drain has new guardrails

The old drain path could run whenever a revision woke the extension. The new path refuses to enter, finish, fail, or reschedule work after the loop has been disabled. 

// desktop/agent.ts
async function drain() {
  if (
    !configured ||
    !session ||
    !host.loop.isEnabled() ||
    draining ||
    session.isStreaming
  ) {
    return;
  }
  ...
  try {
    await session.prompt(...);
    if (!host.loop.isEnabled()) return;
    const continueRunning = host.loop.finishCycle();
    ...
  } catch (error) {
    if (!host.loop.isEnabled()) return;
    host.loop.fail(message);
    ...
  } finally {
    draining = false;
  }
  if (host.loop.isEnabled() && host.loop.snapshot().status === "working") {
    setTimeout(() => void drain(), 0);
  }
}

This is the part that prevents a stopped cycle from becoming an error report or automatically starting the next cycle after cancellation.

Invariants future changes should preserve

InvariantEvidence in this diffWhy it matters
Every app launch begins stopped when Pi is configured. ScribeLoopState constructor sets status: "stopped" and enabled = false. Editing or importing should not invoke the provider until the writer asks.
Stopped revisions are retained. revision() updates latestRevision and latestDocumentRevision even when disabled. Starting after several edits should review the latest durable state, not an old state or every intermediate state.
Abort does not consume a cycle. stop() decrements cycleCount if cycleInProgress. A user cancellation should not push the loop toward the five-cycle cap.
Successful yielded work is not repeated by stop/start. start() chooses waiting when latestRevision <= yieldedRevision. Stopping after a clean yield should not cause a duplicate provider call for unchanged work.

Check your mental model

1. A writer imports three sources while the agent is stopped, then clicks Start Agent. Which revision should be reviewed?
The latest durable project revision. revision() keeps replacing latestRevision while stopped, and main.ts also fetches agent.seed during Start before sending agent.start.
2. What would break if beginCycle() did not check enabled?
A stopped loop could still start provider work when a revision wakes the extension or when drain is scheduled. That would violate the new launch-stopped behavior.
3. Why does stop() decrement cycleCount when a cycle is in progress?
Because the cycle was cancelled by the writer before it completed. Counting it would make repeated cancellations look like failed autonomous attempts and could move the loop toward capped.
4. Where would you start debugging if the Start Agent button is disabled even though Pi is configured?
First check the runtime status passed to SuggestionDock. The button is disabled only for status === "offline" or when controlPending is set. If status is offline, inspect agent configuration diagnostics in Activity and the runtime emitted by desktop/agent.ts.
5. After a successful wait_for_changes, what should Stop then Start do for the same unchanged revision?
It should remain waiting and avoid another provider call. The loop stores yieldedRevision, and start() sets waiting when the latest revision is not newer than the yielded one.
