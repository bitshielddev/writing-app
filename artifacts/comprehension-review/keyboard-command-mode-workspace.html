A command mode has been wired through the writing workspace

This change adds a fixed Ctrl+; keyboard layer, moves layout ownership out of App.tsx, and gives the suggestion dock an externally controlled target so keyboard movement and visual focus agree.

The path a shortcut now takes

1. CaptureuseKeybindingController listens on document during capture and only arms on Ctrl+;.
2. MatchsequenceMatcher compares the typed continuation against DEFAULT_KEYMAP.
3. AdaptuseWorkspaceKeybindings translates a semantic command into editor, layout, inbox, or suggestion-navigation operations.
4. ReflectKeybindingCommandStrip, SuggestionDock, and KeybindingHelpDialog expose pending commands, target movement, and reference help.

Why this had to touch layout

The shortcut handlers need operations such as "open the writing partner", "focus project navigation", and "close drawers before returning to the editor". Those operations were previously spread through App.tsx. The new useWorkspaceLayout hook centralizes panel and drawer state, refs, width persistence, and breakpoint handling so keyboard commands can call the same operations as header buttons.

Evidence: App.tsx removes inline width/drawer code and now destructures the layout controller returned by useWorkspaceLayout.

The keymap is documented and executable from one source

The binding table is data, not hard-coded in the dialog. The help dialog renders COMMAND_CATALOG with DEFAULT_KEYMAP, while the controller matches against that same map.

export const DEFAULT_KEYMAP = {
  "help.open": ["?"],
  "region.navigation.focus": ["h"],
  "region.partner.focus": ["l"],
  "region.editor.focus": ["i"],
  "region.navigation.toggle": ["H"],
  "region.partner.toggle": ["L"],
  "suggestion.next": ["j"],
  "suggestion.previous": ["k"],
  "suggestion.open": ["Enter"],
  "suggestion.back": ["b"],
  "suggestion.pin.toggle": ["p"],
  "suggestion.preview": ["v"],
  "suggestion.dismiss": ["d", "d"],
} as const satisfies Keymap;

The double d for dismiss is deliberate in the code and covered by tests.

Partial sequences are first-class

matchSequence returns partial, exact, or invalid. This lets the command strip display the next valid keys before a command has executed.

const matches = Object.entries(keymap).filter(([, sequence]) =>
  isPrefix(strokes, sequence),
) as [AppCommandId, KeySequence][];

if (!matches.length) return { status: "invalid" };

const exact = matches.find(([, sequence]) => sequence.length === strokes.length);
if (exact) return { status: "exact", commandId: exact[0] };
Maintenance note. Exact matches currently win before longer continuations. Do not add both ["d"] and ["d", "d"] unless the matcher is changed, because the one-stroke command would run immediately.

Where normal typing is protected

The controller only arms when Ctrl+; is pressed without Alt, Meta, Shift, or repeat. After arming, it ignores modifier combinations and repeat keys. If a continuation is invalid, it cancels and shows "Unknown shortcut" before calling preventDefault, so the typed character is not consumed.

if (isLeader(event)) {
  event.preventDefault();
  event.stopPropagation();
  setMessage(undefined);
  setPending(true);
  setSequence([]);
  return;
}

if (!pending) return;

const stroke = normalizeStroke(event);
if (!stroke) {
  cancelSequence();
  return;
}

const nextSequence = [...sequence, stroke];
const match = matchSequence(nextSequence, DEFAULT_KEYMAP);
if (match.status === "invalid") {
  cancelSequence();
  showMessage("Unknown shortcut");
  return;
}

This supports the documentation claim that ordinary typing remains unchanged unless a valid pending sequence is being completed.

Suggestion movement has a separate target from selection

The new navigation hook keeps a target ID for the queue. If a detail page is open, selectedId becomes the effective target. The ordering is explicit: pinned entries first, then live entries.

const orderedEntries = useMemo(
  () => [...pinnedEntries, ...entries],
  [entries, pinnedEntries],
);

const effectiveTargetId =
  selectedId ??
  (targetId && orderedEntries.some((entry) => entry.item.id === targetId)
    ? targetId
    : undefined);

When movement starts without a current target, j lands on the first item and k lands on the last. Boundaries return a status instead of wrapping.

The dock now renders keyboard state instead of owning tab state

SuggestionDock no longer owns the Suggestions/Activity tab itself. App owns partnerView so keyboard commands can force the visible tab to Suggestions before moving or opening an item.

view={partnerView}
keyboardTargetId={
  keyboardNavigationActive ? suggestionNavigator.targetId : undefined
}
regionRef={regionRef}
onViewChange={setPartnerView}
onKeyboardTargetChange={suggestionNavigator.setTargetId}

Inside the dock, the target ID maps to an open button ref. The dock only focuses a row when the queue is visible and detail is not open.

useEffect(() => {
  if (!keyboardTargetId || selectedEntry || view !== "suggestions") return;
  const row = rowRefs.current.get(keyboardTargetId);
  row?.focus();
  row?.scrollIntoView?.({ block: "nearest" });
}, [keyboardTargetId, selectedEntry, view]);

Command handlers are where the invariants live

useWorkspaceKeybindings is the safety layer. It checks whether a command makes sense before mutating inbox or editor state, then returns either executed() or unavailable(message). The status message is shown in the same command strip.

"suggestion.preview": () => {
  if (!targetEntry) return unavailable("No suggestion selected");
  if (!isTextSuggestion(targetEntry.item)) {
    return unavailable("This suggestion cannot be previewed in the document");
  }
  if (targetEntry.withdrawn) {
    return unavailable("This suggestion was withdrawn");
  }
  if (activePreviewId) return unavailable("Finish the active preview first");
  onPreview(targetEntry.item);
  return executed();
},
"suggestion.dismiss": () => {
  if (!targetEntry) return unavailable("No suggestion selected");
  const id = targetEntry.item.id;
  if (activePreviewId === id) {
    return unavailable("Finish this suggestion's preview first");
  }
  const neighborId = navigator.neighborAfterRemoval(id);
  const keepDetailOpen = selectedEntry?.item.id === id && Boolean(neighborId);
  onDismiss(id);
  navigator.setTargetId(neighborId);
  if (keepDetailOpen && neighborId) onSelect(neighborId);
  return executed();
},

The important invariant is preserved: an active preview blocks starting another preview and blocks dismissing that preview's source item.

Modal focus is now shared

The drawer focus logic was extracted into useModalFocus and reused by the shortcut reference dialog. It records the previously focused element, moves initial focus on the next animation frame, traps Tab inside known focusable elements, handles Escape, and restores focus on cleanup.

const previouslyFocused = document.activeElement as HTMLElement | null;
const focusFrame = window.requestAnimationFrame(() =>
  initialFocusRef.current?.focus(),
);

if (event.key === "Escape") {
  event.preventDefault();
  event.stopPropagation();
  onClose();
  return;
}

Because the keybinding controller is disabled while help is open, Escape goes to the modal close path rather than the shortcut-sequence cancel path.

Layout state has a named owner now

The extracted hook preserves the previous responsive contract while making it callable from shortcuts. Desktop toggles affect panels. Below desktop, opening one drawer closes the other. Crossing into desktop closes drawers.

const openContext = useCallback(() => {
  if (isDesktop()) {
    setContextPanelOpen(true);
  } else {
    setContextDrawerOpen(true);
    setNavigationDrawerOpen(false);
  }
}, [isDesktop]);

const closeDrawersAtDesktop = (event: MediaQueryListEvent) => {
  setDesktop(event.matches);
  if (event.matches) closeDrawers();
};
Confirmed by tests. useWorkspaceLayout.test.ts covers below-desktop drawer exclusivity, closing drawers at the desktop breakpoint, and desktop panel toggles that do not open drawers.

Risks worth remembering during maintenance

  • Prefix collisions: the matcher cannot currently represent a command that is both complete and a prefix for a longer command.
  • Global capture: the controller listens at document capture phase. Keep the disabled path correct for modal or editing states that should own keys.
  • Browser globals: useWorkspaceLayout reads window.matchMedia during initial state. This matches the app's no-SSR assumption, but tests must stub it.
  • Target versus selection: queue targeting and selected detail are related but not identical. Future dock changes should keep the target stable across pin/unpin/dismiss transitions.
  • Modal scope: useModalFocus is a local focus trap. The docs still say it does not set background inert, hide the app from assistive tech, or lock body scroll.

Check your mental model

1. Why does the change introduce command IDs instead of binding handlers directly to keys?

The command IDs separate meaning from keystrokes. commands.ts defines stable actions and catalog text; defaultKeymap.ts binds keys; useWorkspaceKeybindings.ts supplies handlers. This lets the help dialog, matcher, and handlers share one model.

2. What happens when the user presses Ctrl+; d?

The controller enters a partial sequence, because ["d"] is a prefix of ["d", "d"]. It does not dismiss yet. The command strip can show "Dismiss suggestion" as the continuation. The second d executes suggestion.dismiss.

3. If detail is open and j moves to another suggestion, why does onSelect run?

useSuggestionKeyboardNavigation treats selectedId as the effective target. In move, after setting the next target, it calls onSelect(nextId) only when selectedId exists. That keeps detail view moving through entries instead of only moving the queue highlight.

4. What would break if the dock kept its own Suggestions/Activity state?

Keyboard commands could not reliably force the Suggestions view before moving or opening a target. App now owns partnerView and passes onViewChange, so both pointer tab clicks and shortcut handlers operate on the same state.

5. Where would you start debugging if Ctrl+; v says a preview is unavailable?

Start in the suggestion.preview handler in useWorkspaceKeybindings.ts. It has the context checks: no target, non-text suggestion, withdrawn entry, or existing activePreviewId. If those pass, then debug handlePreview in App.tsx.
