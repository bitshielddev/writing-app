The suggestion path is now one controller, and the mock injection path is gone

This staged change removes the development-only mock suggestion UI and folds suggestion persistence, optimistic rendering, selection, preview state, and event reconciliation into useSuggestionController. It also narrows the persisted suggestion schema so stale/withdrawn flags stay in the renderer.

70 files touched 854 insertions / 2227 deletions Main history context: document scoping, suggestion history, process resilience

Read this as a boundary change

The main behavior change is not a new suggestion kind or UI button. The important change is where suggestion state lives while the user is interacting with it. Previously, a feed adapter, an inbox reducer, and a persistence hook cooperated. Now the renderer has one suggestion controller that owns the live projection and talks directly to the desktop command API.

The removal of the mock suggestion window matters because it deletes a second suggestion entry point: no scribeDevelopment preload bridge, no development.suggestion.create operation, no development actor, and no /mock-suggestions renderer route remain in this diff.

The three behavior areas to keep in your head

Renderer suggestion orchestration

src/suggestions/useSuggestionController.ts replaces useSuggestionInbox plus useSuggestionPersistence. It holds the authoritative state, optimistic queue, transient detail/preview entries, retry/flush/discard controls, and desktop-event reconciliation.

Process and contract surface

The development suggestion operation is removed from shared contracts, preload, main IPC, storage operations, and the development menu. Persisted inbox and pinned entries no longer accept stale or withdrawn fields.

Docs, tests, and budgets

Tests move away from reducer and mock-controller coverage toward controller reconciliation tests. Documentation now describes committed desktop events and the controller. Bundle checks stop looking for deleted mock-development code.

Follow one writer command through the new controller

The controller keeps two versions of suggestion state: an authoritative projection from storage, and a displayed projection computed by applying active and pending commands. The code that builds that displayed projection is small, but it is the key invariant.

// src/suggestions/useSuggestionController.ts
function optimisticProjection(
  authoritative: PersistedSuggestionState,
  active: Candidate | undefined,
  pending: Candidate[],
) {
  let projection = structuredClone(authoritative);
  for (const candidate of [...(active ? [active] : []), ...pending]) {
    const transition = applySuggestionCommand(projection, candidate.command);
    if (transition.status !== "rejected") projection = transition.state;
  }
  return projection;
}

This means optimistic rendering and storage use the same durable command policy from transitions.ts. Rejected optimistic commands are ignored for the projection instead of being forced into UI state.

The expectedSuggestionRevision line is what prevents a command from being applied against a stale projection without storage noticing. The controller sends one command at a time; after the first response advances revision.current, the next command uses the newer revision.

When storage speaks before the RPC response

Durable events and direct command responses can arrive in either order. The old persistence hook only updated authoritative state on events and responses separately. The new controller explicitly recognizes the command ID on a suggestion.event and can clear the active command from the event path.

// src/suggestions/useSuggestionController.ts
const onDesktopEvent = useCallback((event: DesktopEvent) => {
  if (event.type !== "suggestion.event" ||
      event.suggestionRevision <= revision.current) return;
  updateTransientFromEvent(event.event);
  authoritative.current = structuredClone(event.state);
  revision.current = event.suggestionRevision;
  if (event.commandId && active.current?.commandId === event.commandId) {
    active.current = undefined;
    failed.current = undefined;
    setFailureMessage(undefined);
  }
  publishProjection();
  if (pending.current.length && !active.current) drain();
  else if (!active.current) setStatus({ state: "idle", acknowledgedVersion: revision.current });
}, [drain, publishProjection, updateTransientFromEvent]);

This protects an important race: if storage emits the durable event first, the controller should not apply the command twice when the RPC resolves later. The new test named acknowledges a matching durable event once regardless of RPC ordering covers that path.

Maintenance notes for future changes

Keep transition policy shared

If you add a writer command, update DurableSuggestionCommand, applySuggestionCommand, storage facts in desktop/domain/suggestion-persistence.ts, contract schemas, and controller methods together. The controller assumes the optimistic projection and storage policy are the same policy.

Be careful with failed command state

retry, flush, and discard now live beside selection and preview state. A change to failure handling can affect document switching because useWorkspaceController waits for flushSuggestions() before selecting or creating a document.

Geometry commands are special

enqueue coalesces pending workspace.geometry commands by suggestion ID, and a storage conflict for geometry is retried with a new command ID. Other conflicts move the controller to failed.

Manual suggestion data needs a new route if it returns

The old mock route was removed from contracts, preload, main, renderer, tests, and docs. Reintroducing manual injection would be a cross-boundary feature, not a small UI restore.

Comprehension check

A user pins a suggestion while a previous mark-viewed command is still active. What should the controller display and when should the pin command be sent?

Answer in terms of optimistic projection and revision sequencing.

Answer

The UI can display the pinned state optimistically because optimisticProjection applies the active command plus pending commands to the authoritative state. The pin command should not be sent to storage until the active command resolves or is acknowledged by a matching event. When it is sent, it uses the updated expectedSuggestionRevision.

Why should stale and withdrawn not be written into hydrated suggestion state after this diff?

Answer

The shared contract no longer includes those fields on persisted inbox or pinned entries. They are renderer-only flags attached to transient selected or preview entries. Persisting them would fail strict contract validation and would mix UI recovery state into durable projection state.

What would break if onDesktopEvent did not clear active.current for a matching commandId?

Answer

If the durable event arrives before the RPC response, the controller would still think the command is active. That could block draining pending commands, keep the status in a saving/pending state, or cause later response handling to interact with already-applied state. The new test covers this ordering.

Where would you start if someone asks to bring back manual development suggestions?

Answer

Start at the cross-process contract, not the deleted React form. This diff removed development.suggestion.create, DEVELOPMENT_SUGGESTION_CHANNEL, scribeDevelopment, the main IPC route, the storage operation, the development actor, and the renderer route. A replacement needs an intentional contract and storage path.

Which current risk is tied to DATABASE_VERSION changing from 7 to 1?

Answer

Existing alpha databases with a higher schema version can be treated as too new for the current baseline. The code and docs say incompatible alpha databases are not migrated in place, so local users may need to preserve or delete the old database before continuing.
