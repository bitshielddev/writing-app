Desktop boundary routing walkthrough

This change pulls Electron boundary code out of desktop/main.ts, desktop/preload.ts, desktop/storage.ts, and desktop/agent.ts into smaller transport, routing, and startup modules. Runtime behavior is meant to stay the same, while the process contracts become testable without booting Electron.

The change is mostly a boundary extraction

The old main process owned several jobs in one file: child process RPC, sender validation, IPC handlers, storage event fan-out, agent message handling, startup sequencing, and window creation. The diff keeps the Electron-specific shell in desktop/main.ts and moves the logic into adapters with narrow dependencies.

Main process

desktop/main.ts now creates real Electron objects and delegates to ChildRpc, registerMainIpc, createStorageMessageHandler, createAgentMessageHandler, and startDesktop.

Utility processes

desktop/storage.ts and desktop/agent.ts no longer parse every message inline. They use transport helpers that validate envelopes and correlate results.

Renderer

The workspace controller owns one desktop event subscription and sends suggestion events into an in-memory relay consumed by useSuggestionInbox.

The shared vocabulary is now explicit

The new src/shared/contracts.ts file is the anchor for names that were previously repeated as string literals or local types. This matters because four boundaries need to agree: renderer to preload, preload to main, main to utility children, and agent to storage through main.

export const DESKTOP_INVOKE_CHANNELS = {
  hydrate: "scribe:hydrate",
  startAgent: "scribe:agent.start",
  stopAgent: "scribe:agent.stop",
  saveDocument: "scribe:document.save",
  saveSuggestionState: "scribe:suggestions.save",
  importSource: "scribe:source.import",
} as const satisfies Record<Exclude<keyof DesktopBridge, "subscribe">, string>;

export const DESKTOP_EVENT_CHANNEL = "scribe:event" as const;
export const DEVELOPMENT_SUGGESTION_CHANNEL =
  "scribe:development.suggestion.create" as const;

The satisfies clause creates a useful invariant: if DesktopBridge gains another invoke method, TypeScript should force the channel inventory to change too. The subscription method is excluded because it uses the event channel instead of ipcRenderer.invoke.

export type ChildMessage =

The contract tests do not prove every runtime payload is valid. They do make the inventories visible and fail if someone changes the channel map or message-kind list without acknowledging the boundary change.

Follow the two runtime paths

Most behavioral claims in this diff are about preserving existing flow while moving code. The best way to read it is by following the messages.

Path A: renderer calls main

preloadcreateDesktopBridge maps each renderer method to a shared invoke channel.
main IPCregisterMainIpc validates the sender before calling storage or agent RPC.
child processChildRpc.call waits for readiness, assigns an ID, sends { kind: "rpc" }, and resolves only the matching result.
const register = (
  channel: string,
  handler: (event: MainInvokeEvent, ...args: unknown[]) => unknown,
) => {
  ipcMain.handle(channel, (event, ...args) => {
    if (!validateSender(event.sender)) throw new Error("Unknown renderer");
    return handler(event, ...args);
  });
};

register(DESKTOP_INVOKE_CHANNELS.startAgent, async () => {
  const seed = await storage.call<ObservationSeed>("agent.seed");
  return agent.call<AgentRuntime>("agent.start", {
    projectRevision: seed.projectRevision,

The sender check is still on every route because the register helper wraps every handler. Starting the agent still uses storage as the source of the observation seed, then passes revisions to agent.start.

async call<T>(method: string, params?: unknown): Promise<T> {
  await this.ready;
  if (this.disposed) throw new Error("Utility process is not available");
  const id = this.createId();
  return new Promise<T>((resolve, reject) => {
    this.pending.set(id, {
      resolve: (value) => resolve(value as T),
      reject,
    });
    const request: RpcRequest = { kind: "rpc", id, method, params };
    this.child.postMessage(request);
  });
}

The behavior to notice is correlation rather than ordering. Tests send results out of order and expect the right promises to resolve.

Path B: child processes notify main

Storage domain events still get broadcast to renderers. They also inform the agent that the project revision changed. The document path can use the event's document revision directly; the source-import path asks storage for a fresh seed.

if (message.kind !== "domain.event") return;
broadcast(message.event);
if (message.event.type === "document.saved") {
  getAgent()?.post({
    kind: "project.changed",
    projectRevision: message.event.projectRevision,
    documentRevision: message.event.document.revision,
  });
} else if (message.event.type === "source.imported") {
  const seed = await storage.call<ObservationSeed>("agent.seed");
  getAgent()?.post({
    kind: "project.changed",
    projectRevision: seed.projectRevision,
    documentRevision: seed.documentRevision,
  });
}

Agent-originated storage requests are still proxied through main. A storage failure becomes a storage.result with an error string rather than an unhandled rejection.

if (message.kind === "storage.request") {
  try {
    const result = await storage.call(message.method, message.params);
    getAgent()?.post({ kind: "storage.result", id: message.id, result });
  } catch (error) {
    getAgent()?.post({
      kind: "storage.result",
      id: message.id,
      error: error instanceof Error ? error.message : String(error),
    });
  }
} else if (message.kind === "agent.runtime") {
  setRuntime(message.runtime);
} else if (message.kind === "agent.activity") {
  broadcast({ type: "agent.activity", activity: addActivity(message.activity) });
}
A subtle dependency remains: these helpers use getAgent()?.post(...). If an event arrives before the agent object exists, it is dropped. That matches the defensive optional chaining in the changed code, but it is an assumption to remember when changing startup order.

Startup order became a named behavior

desktop/startup.ts extracts the sequence that was previously embedded in main.ts. The sequence is now small enough to test without Electron.

const storage = spawnStorage();
await storage.ready;
await storage.call("workspace.repair");

const agent = spawnAgent();
await agent.ready;

registerIpc(storage, agent);
installMenu();
createWindow();

const seed = await storage.call<ObservationSeed>("agent.seed");
agent.post({
  kind: "project.changed",
  projectRevision: seed.projectRevision,
  documentRevision: seed.documentRevision,
});

The invariant is direct: repair happens before the agent starts, IPC is registered only after both children are ready, and the agent gets an initial project.changed message after the window is created. The tests assert this call order.

The renderer now has one desktop subscription

Before this change, the desktop suggestion feed wrapped bridge.subscribe and filtered suggestion.event inside the feed. The workspace controller also subscribed to desktop events for document, source, runtime, and activity updates. That could create multiple bridge-level subscriptions inside one controller.

export function createSuggestionFeedRelay(): {
  feed: SuggestionFeed;
  emit: (event: SuggestionEvent) => void;
} {
  const listeners = new Set<(event: SuggestionEvent) => void>();
  return {
    feed: {
      subscribe(listener) {
        listeners.add(listener);
        return () => listeners.delete(listener);
      },
    },
    emit(event) {
      listeners.forEach((listener) => listener(event));
    },
  };
}

The controller creates the relay once and passes relay.feed to the inbox. Its single desktop subscription handles every event type. Suggestion events are forwarded into the relay.

const suggestionFeed = useMemo(() => createSuggestionFeedRelay(), []);
const feed = suggestionFeed.feed;
const inbox = useSuggestionInbox(feed, inboxOptions);

useEffect(
  () =>
    desktop.subscribe((event) => {
      switch (event.type) {
        case "document.saved":
          documentRevisionRef.current = event.document.revision;
          break;
        case "source.imported":
          setSources((current) => [
            event.source,
            ...current.filter((source) => source.id !== event.source.id),
          ]);
          break;
        case "agent.runtime":
          setRuntime(event.runtime);
          break;
        case "agent.activity":
          setActivity((current) => upsertActivity(current, event.activity));
          break;
        case "suggestion.event":
          suggestionFeed.emit(event.event);
          break;
      }
    }),
  [desktop, suggestionFeed],
);
The docs were updated to match this mental model: feed subscribers now live in createSuggestionFeedRelay, while useWorkspaceController owns the desktop subscription.

Maintenance notes for the next change

  • If a renderer bridge method is added, update DESKTOP_INVOKE_CHANNELS, preload-bridge.ts, ipc-routing.ts, and the contract/preload/IPC tests together.
  • If a child message kind is added, update ChildMessage, CHILD_MESSAGE_KINDS, and whichever handler should receive it. ChildRpc currently accepts by kind, then trusts the shape of most payloads.
  • If agent.start changes its params, update createAgentParentTransport. It currently rejects malformed start messages unless params contains integer projectRevision and documentRevision.
  • If suggestion events stop appearing in the UI, check the controller's case "suggestion.event": suggestionFeed.emit(event.event) first. The inbox no longer subscribes to the desktop bridge directly.
  • The generated docs HTML and search index changed alongside the markdown docs. Treat them as derived artifacts unless this project expects docs HTML to be reviewed directly.

Comprehension check

1. When the renderer calls "start agent", where do the project and document revisions come from?

They come from storage via storage.call("agent.seed") inside registerMainIpc. Main then calls agent.call("agent.start", { projectRevision, documentRevision }).

2. What would break if the workspace controller stopped calling suggestionFeed.emit for suggestion.event?

The inbox would not receive live suggestion additions, updates, or retractions. After this change, useSuggestionInbox subscribes to the relay feed, not to the desktop bridge.

3. Why does ChildRpc store requests by ID instead of resolving calls in send order?

Utility process responses can arrive out of order. The ID map lets rpc.result resolve or reject only the matching pending promise.

4. Where would you start debugging if a source import updates the UI but the agent does not observe the new source?

Start in createStorageMessageHandler. A source.imported event should be broadcast, then main should call storage.call("agent.seed") and post { kind: "project.changed", ... } to the agent.

5. Which invariant does the startup test protect?

Storage must be ready and repaired before the agent starts; IPC/menu/window setup happens after both children are ready; then main sends the initial project.changed seed to the agent.
