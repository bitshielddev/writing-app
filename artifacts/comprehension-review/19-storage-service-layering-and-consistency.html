Storage is now a service with sharper failure boundaries

This diff turns desktop/storage.ts from a 503-line process script with embedded persistence logic into a thin Electron worker entrypoint. The storage behavior now lives behind createStorageService, with repositories, file operations, transactions, RPC mapping, and event dispatch separated enough to test failure cases directly.

The short version, grounded in the diff

What moved

Most of the old top-level functions in desktop/storage.ts moved into files under desktop/storage/. The old entrypoint now only validates startup arguments, creates a service, wires the transport, and starts pending event dispatch.

What changed behaviorally

Document saves are serialized and re-check the expected revision inside the database transaction. If the draft file is written but the database transaction fails, the draft mirror is repaired from SQLite.

What became testable

The service accepts injected workspace file handlers, event publishers, loggers, and workspace roots. The new layer tests simulate disk failures, database trigger failures, publisher failures, and concurrent saves.

What changed for callers

The previous test exports handleStorageRequest, getWorkspacePathsForTest, and closeStorageForTest are gone. Tests now create isolated StorageService instances.

The new storage shape

The main design move is dependency assembly in one place. createStorageService opens the database, bootstraps rows, constructs repositories, builds file and event adapters, then exposes a narrow service object.

Entrypointdesktop/storage.ts handles process startup and transport.
Service factoryservice.ts owns composition and cleanup on construction failure.
Operationsoperations.ts contains storage use cases.
AdaptersRepositories, outbox, and workspace files wrap SQLite, publishing, and disk IO.
export function createStorageService(options: CreateStorageServiceOptions) {
  const projectId = options.projectId ?? DEFAULT_PROJECT_ID;
  const documentId = options.documentId ?? DEFAULT_DOCUMENT_ID;
  const paths = createStoragePaths(options.workspaceRoot);
  const database = new SqliteDatabaseLifecycle(options.databasePath);
  try {
    database.run(() => bootstrapWorkspace(database.db, projectId, documentId));
    const projects = new ProjectRepository(database.db);
    const documents = new DocumentRepository(database.db);
    const sources = new SourceRepository(database.db);
    const suggestions = new SuggestionRepository(database.db);
    const outbox = new OutboxRepository(database.db);
    const files = options.createWorkspaceFiles?.(paths) ?? new NodeWorkspaceFiles(paths);
    const publisher: EventPublisher = {
      publish: options.publishEvent ?? (() => undefined),
    };
    const dispatcher = new OutboxDispatcher(outbox, publisher);
    const operations = new StorageOperations({
      projectId,
      documentId,
      paths,
      transactions: database,
      projects,
      documents,
      sources,
      suggestions,
      outbox,
      dispatcher,
      files,
      logger: options.logger,
    });
    return {
      paths,
      database,
      operations,
      handleRequest: createStorageRequestHandler(operations),
      dispatchPendingEvents: () => dispatcher.dispatch(),
      close: () => database.close(),
    };
  } catch (error) {
    database.close();
    throw error;
  }
}

Two details matter for future maintainers:

  • The service can be created without process.parentPort, so tests no longer rely on module-level globals or a shared in-memory database.
  • The catch in the factory closes the database if construction fails after opening SQLite. That is a lifecycle invariant, not just cleanup style.

The save path now defends both revision and mirror consistency

The document save path is the most important behavioral change. The old code checked the expected revision before writing draft.md, then updated SQLite. The new code queues document saves, writes the draft, checks the revision again inside the transaction, and repairs the draft if the transaction fails.

async saveDocument(params: unknown): Promise<DocumentSnapshot> {
  const previous = this.documentOperation;
  let release!: () => void;
  this.documentOperation = new Promise<void>((resolve) => {
    release = resolve;
  });
  await previous;
  try {
    return await this.performDocumentSave(params);
  } finally {
    release();
  }
}

This queue is only for document saves. It stops two concurrent requests from both using the same revision after they have both passed validation. The second request still runs, but by then the persisted document revision has changed, so it should fail with DOCUMENT_REVISION_CONFLICT.

await this.deps.files.writeDraft(input.markdown);
let saved: DocumentSnapshot;
try {
  saved = this.deps.transactions.run(() => {
    if (input.expectedRevision !== this.deps.documents.get(this.deps.documentId).revision) {
      throw new Error("DOCUMENT_REVISION_CONFLICT");
    }
    const now = Date.now();
    const document = this.deps.documents.save(
      this.deps.documentId,
      input.blocks as unknown[],
      input.markdown as string,
      now,
    );
    this.deps.projects.incrementRevision(this.deps.projectId, now);
    this.deps.outbox.enqueue({
      type: "document.saved",
      document,
      projectRevision: this.deps.projects.get(this.deps.projectId).revision,
    });
    return document;
  });
} catch (error) {
  try {
    await this.deps.files.repairDraft(
      this.deps.documents.get(this.deps.documentId).markdown,
    );
  } catch (repairError) {
    throw new AggregateError(
      [error, repairError],
      "Document save failed and draft repair failed",
      { cause: repairError },
    );
  }
  throw error;
}
await this.deps.dispatcher.dispatch();
return saved;

Follow the state: on a successful save, draft.md, the documents row, project revision, and the event outbox advance together. On a transaction failure after the draft write, the database remains authoritative and repairDraft pulls the file mirror back to the database markdown.

Events are still durable, but dispatch is now an explicit collaborator

The old storage file queued events and flushed them from module-level code. The extracted version makes the outbox a repository and gives dispatch its own small class.

export class OutboxDispatcher {
  private activeDispatch: Promise<void> = Promise.resolve();

  constructor(
    private readonly repository: EventOutbox,
    private readonly publisher: EventPublisher,
  ) {}

  dispatch() {
    const dispatch = this.activeDispatch.then(() => this.deliverPending());
    this.activeDispatch = dispatch.catch(() => undefined);
    return dispatch;
  }

  private async deliverPending() {
    for (const row of this.repository.pending()) {
      await this.publisher.publish(row.event);
      this.repository.markDispatched(row.sequence);
    }
  }
}

Marking happens after publish, so a thrown publisher leaves the row pending. The dispatcher serializes overlapping dispatch calls through activeDispatch. Startup also dispatches pending events after the worker announces readiness:

await service.operations.repairWorkspace();
const receive = createStorageTransport(
  service.handleRequest,
  (message) => process.parentPort?.postMessage(message),
);
process.parentPort?.on("message", ({ data }: { data: unknown }) => {
  void receive(data);
});
process.once("exit", () => service.close());
process.parentPort?.postMessage({ kind: "ready" });
await service.dispatchPendingEvents();

Operational consequence: methods such as document.save, source.import, and suggestion mutations now await dispatch after committing. The layer test confirms that a publisher failure rejects the request while leaving the committed database change and pending outbox row intact. Consumers need to treat a rejected request after commit as possible for event-publishing failures.

RPC surface preserved, test surface changed

The desktop RPC method names are still centralized in one map and checked against StorageRpcMethod. Unknown methods still throw the same shape of error.

export function createStorageRequestHandler(operations: StorageOperations) {
  const operationMap = {
    hydrate: () => operations.hydrate(),
    "workspace.repair": () => operations.repairWorkspace(),
    "document.save": (params) => operations.saveDocument(params),
    "suggestions.save": (params) => operations.saveSuggestionState(params),
    "source.import": (params) => operations.importSource(params),
    "agent.seed": () => operations.getObservationSeed(),
    "agent.suggestions.list": () => operations.listSuggestions(),
    "agent.suggestion.create": (params) => operations.createSuggestion(params),
    "agent.suggestion.update": (params) => operations.updateSuggestion(params),
    "agent.suggestion.retract": (params) => operations.retractSuggestion(params),
    "development.suggestion.create": (params) => operations.createDevelopmentSuggestion(params),
  } satisfies Record<StorageRpcMethod, Operation>;

  return async (method: string, params?: unknown) => {
    const operation = operationMap[method as StorageRpcMethod] as Operation | undefined;
    if (!operation) throw new Error(`Unknown storage method: ${method}`);
    return await operation(params);
  };
}

What future you should keep in mind

  • Commit-then-dispatch can reject after persistence. Because operations await dispatcher.dispatch() after the transaction, a caller may see an error even though the database commit happened. This is intentional as tested, but it affects retry semantics.
  • Only document saves are serialized. Suggestion mutations and source imports still rely on their transaction boundaries and revision checks. If future work introduces cross-operation invariants, this queue will not protect them automatically.
  • suggestions.save does not dispatch events. That matches the moved behavior: it updates the projection after validating items and trimming entries. If a later caller expects domain events from projection replacement, that would be a new requirement.
  • File writes remain outside SQLite transactions. The code compensates with ordering and repair, but there is no true atomic transaction across disk and SQLite. The tests now document the chosen compensation paths.
  • Entrypoint side effects changed. Importing desktop/storage.ts no longer opens an in-memory database unless the module is running as a parent-port worker. Tests and tools should use createStorageService.

Small glossary for this diff

Draft mirrorThe workspace file draft.md. SQLite remains the durable source for document state; the mirror is repaired from the database when needed.
OutboxThe event_outbox table. Events are inserted during the same transaction as the domain change, then dispatched later and marked with dispatched_at.
Storage serviceThe object returned by createStorageService: paths, database lifecycle, operations, request handler, pending-event dispatch, and close.
Workspace files adapterThe injectable boundary around disk operations such as draft writes, source copies, and source cleanup.

Comprehension check

Use these as quick review prompts before modifying this area again.

1. Two document.save requests arrive with the same expected revision. What should happen now?

Only one should fulfill. saveDocument serializes document saves and performDocumentSave re-checks the expected revision inside the transaction. After the first save increments the document revision, the second request should fail with DOCUMENT_REVISION_CONFLICT.

2. Why does the code repair draft.md in the catch block after writeDraft?

The draft file is written before the SQLite transaction. If SQLite then rejects the update, the file could contain markdown that is not in the database. The repair step writes the current database markdown back to the draft mirror.

3. A publisher throws while dispatching a committed document.saved event. What state should you expect?

The document save has already committed, the request rejects because dispatch failed, and the outbox row remains pending because markDispatched happens only after a successful publish. A later dispatchPendingEvents() can retry it.

4. What would break if tests kept importing handleStorageRequest from desktop/storage.ts?

That export no longer exists. The new test boundary is createStorageService, which avoids module-level storage state and lets each test own its workspace root, database, file adapter, and close lifecycle.

5. Where would you start debugging if source import copies a file but the database insert fails?

Start in StorageOperations.importSource. It calls files.copySource, then inserts in a transaction. If that transaction fails, it attempts files.removeSource(copied.destination); if cleanup fails too, it logs through the injected logger and rethrows the original database failure.
