The main move: from a custom observer to a Pi project session

The previous runtime used @earendil-works/pi-agent-core, app-owned agent.yaml, extracted source text in SQLite, per-run transcript tables, and a scheduler-style observation path. The new runtime uses @earendil-works/pi-coding-agent, Pi's own settings/auth/model/session managers, a Scribe extension, and filesystem tools limited to reading the app-managed project workspace.

The managed workspace is now part of the data model

The database schema gained a markdown column on documents and changed sources from extracted text to copied Markdown files with byte counts. Storage now receives a project workspace path as a process argument.

const sourcesDirectory = join(workspaceRoot, "sources");
const piDirectory = join(workspaceRoot, ".pi");
const draftPath = join(workspaceRoot, "draft.md");

The docs describe this as <userData>/projects/default-project/ containing draft.md, sources/, and .pi/sessions/. The code above is the storage-side boundary that makes those paths real.

The agent's read path is simpler now. Instead of reading source text extracted into SQLite, Pi reads the same managed files that storage repairs and owns.

Startup repairs draft.md from SQLite

async function repairDraftMirror() {
  await mkdir(sourcesDirectory, { recursive: true });
  await mkdir(piDirectory, { recursive: true });
  const markdown = getDocument().markdown;
  let current: string | undefined;
  try {
    current = await readFile(draftPath, "utf8");
  } catch (error) {
    if ((error as NodeJS.ErrnoException).code !== "ENOENT") throw error;
  }
  if (current !== markdown) await atomicWrite(draftPath, markdown);
  return { workspaceRoot, draftPath, sourcesDirectory, repaired: current !== markdown };
}

This makes SQLite the canonical source for the accepted draft while keeping a file mirror current for Pi. Storage also calls await repairDraftMirror(); during module startup and main calls workspace.repair before the agent starts.

Source import has narrowed to UTF-8 Markdown

function validateMarkdownSource(path: string, bytes: Uint8Array) {
  const extension = extname(path).toLocaleLowerCase();
  if (extension !== ".md" && extension !== ".markdown") {
    throw new Error("Only .md and .markdown source files are supported");
  }
  try {
    new TextDecoder("utf-8", { fatal: true }).decode(bytes);
  } catch {
    throw new Error("Markdown source must contain valid UTF-8");
  }
}

This is a real product behavior change. PDF, DOCX, JSON, and plain text extraction code has been removed along with mammoth, pdf-parse, and canvas dependencies. A source must now be readable Markdown.

The loop decides when the agent should work, wait, or stop retrying

The new loop is isolated in desktop/scribe-loop.ts. It tracks the latest durable project revision, the active cycle, yielded revision, cycle count, status, and error. That gives the extension a small state machine around a long-lived Pi session.

beginCycle() {
  if (this.state.latestRevision < 0) return undefined;
  if (this.state.latestRevision <= (this.state.yieldedRevision ?? -1)) {
    this.state.status = "waiting";
    return undefined;
  }
  if (this.state.cycleCount >= 5) {
    this.state.status = "capped";
    return undefined;
  }
  this.state.activeRevision = this.state.latestRevision;
  this.state.activeDocumentRevision = this.state.latestDocumentRevision;
  this.state.cycleCount += 1;
  this.state.status = "working";
  this.yieldedInCycle = false;
  return {
    projectRevision: this.state.activeRevision,
    documentRevision: this.state.activeDocumentRevision,
    cycleCount: this.state.cycleCount,
  };
}

The cap is the guard against an agent that never calls wait_for_changes. The tests confirm five consecutive cycles without a yield or newer revision produce capped.

The extension is the bridge between Pi and Scribe storage

pi.registerTool({
  name: "create_suggestion",
  label: "Create suggestion",
  description: "Publish a proposed draft change without editing draft.md.",
  parameters: suggestionSchema,
  execute: async (_id, params) => {
    const item = toSuggestion(params as SuggestionInput);
    const expectedDocumentRevision = host.loop.snapshot().activeDocumentRevision;
    if (expectedDocumentRevision === undefined) return toolResult("No active revision", true);
    try {
      return toolResult(await host.storageCall("agent.suggestion.create", {
        item,
        expectedDocumentRevision,
      }));
    } catch (error) {
      host.wake();
      return toolResult(error instanceof Error ? error.message : String(error), true);
    }
  },
});

Every suggestion mutation carries the active document revision. Storage rejects stale mutations with STALE_SUGGESTION_REVISION. The extension then wakes the loop so the agent can work against the newest revision instead of publishing against an old draft.

The developer can now inspect what the agent did during this launch

Agent events are no longer stored as durable transcript rows. Instead, main holds a launch-scoped ring of redacted activity entries and broadcasts updates to the renderer.

export class ActivityRing {
  private items: AgentActivity[] = [];

  add(input: Omit<AgentActivity, "updatedAt" | "payload"> & { payload?: unknown }) {
    const activity: AgentActivity = {
      ...input,
      updatedAt: Date.now(),
      ...(input.payload === undefined ? {} : { payload: safeActivityPayload(input.payload) }),
    };
    const existing = this.items.findIndex((item) => item.id === activity.id);
    if (existing >= 0) {
      this.items[existing] = {
        ...this.items[existing],
        ...activity,
        timestamp: this.items[existing].timestamp,
      };
    } else {
      this.items.push(activity);
      if (this.items.length > MAX_ITEMS) this.items.splice(0, this.items.length - MAX_ITEMS);
    }
    return existing >= 0 ? this.items[existing] : this.items.at(-1)!;
  }
}

Two details are worth preserving if this area changes later: streaming events aggregate by ID, and the original timestamp is retained when an entry is updated. The tests cover both aggregation and the 500-item cap.

What future maintainers should keep in their head

Schema versioning currently resets data. When PRAGMA user_version differs from DATABASE_VERSION, storage drops the old tables before creating version 2. That is fine for a development reset, but it is not a preserving migration.

if (currentVersion.user_version !== DATABASE_VERSION) {
  db.exec(`
    DROP TABLE IF EXISTS agent_transcript;
    DROP TABLE IF EXISTS agent_runs;
    DROP TABLE IF EXISTS agent_memory;
    DROP TABLE IF EXISTS provider_settings;
    DROP TABLE IF EXISTS event_outbox;
    DROP TABLE IF EXISTS suggestion_state;
    DROP TABLE IF EXISTS sources;
    DROP TABLE IF EXISTS documents;
    DROP TABLE IF EXISTS projects;
    DROP TABLE IF EXISTS app_meta;
  `);
}
  • Keep SQLite and draft.md synchronized. saveDocument writes the Markdown file before the database transaction. If the DB commit fails after the file write, startup repair should restore the mirror from SQLite.
  • Do not widen Pi tools without revisiting the safety model. The system prompt says never edit project files, but the hard boundary is the active tool list and excludeTools.
  • Suggestion mutations must stay revision-checked. Removing expectedDocumentRevision would let stale cycles publish against a draft the user has already changed.
  • Activity payloads can include provider requests/responses. safeActivityPayload redacts sensitive keys and truncates large payloads; new activity producers should keep sending raw payloads through the ring rather than bypassing it.
  • Docs and generated HTML were updated to match the runtime. The behavior should be verified in source files first, then docs.

Comprehension check

Use these to check whether you can safely modify this code path later.

1. A user edits the draft. Which saved artifact does Pi read, and where is it produced?

Pi reads draft.md in the managed workspace. The renderer sends lossy Markdown with desktop.saveDocument, and storage writes that Markdown through atomicWrite(draftPath, markdown) before committing the new document/project revision.

2. Why do Scribe suggestion tools include expectedDocumentRevision?

It ties a create/update/retract request to the revision the active agent cycle reviewed. Storage rejects mismatches with STALE_SUGGESTION_REVISION, preventing stale suggestions from being applied after the user changes the draft.

3. What happens if the agent completes five cycles for the same revision without calling wait_for_changes?

ScribeLoopState sets status to capped and stops immediately starting another cycle. A newer project revision resets the cycle count and can wake the loop again.

4. Where would you start debugging if suggestions stop arriving but the model is still sending messages?

Open the Activity view first to inspect message/tool/error/provider events from the launch ring. Then check whether tool calls are returning stale revision errors, whether runtime status is waiting, capped, or error, and whether the active tool set passed the diagnostics in desktop/agent.ts.

5. What behavior changed for imported source files?

Imports now accept only .md and .markdown files that decode as UTF-8. The file is copied into sources/ with a readable collision-safe filename, and storage records byte count rather than extracted text.
