Faster contracts and activity rendering

This change removes full document snapshots from save responses and save events, compiles reusable contract validators, and reduces the work done when hundreds of agent activity items reach the renderer. It also tightens the agent's model of canonical BlockNote data.

Base: 30dfdae 30 staged files 251 additions / 92 deletions Reviewed 2026-07-12

The change in one pass

The staged work has two performance paths and one related correctness clarification. The save path now transports a five-field receipt after persistence instead of echoing the complete BlockNote document. Contract schemas are compiled once per schema object and cached. Agent activity uses indexed batch upserts and memoized React rows. New benchmarks exercise a 234 KB contract value and a 500-item renderer burst.

Save boundary

src/contracts/events.ts, both operation registries, storage operations, autosave, the main message adapter, and their test fixtures adopt DocumentSaveReceipt.

Validation cost

src/contracts/validation.ts replaces interpreted Check/Errors calls with cached TypeBox validators. scripts/measure-contracts.mjs records timing.

Renderer burst cost

useAgentController.ts indexes IDs once per flush. AgentActivityList.tsx memoizes sorting, the list, and each row. The E2E-only bridge injects 500 activities.

Agent document meaning

The Pi system prompt and a storage test state that blocks retains canonical structure while plainTextBlocks is a helper for supported text anchors.

Scope boundary

The save request still contains the full blocks array. The optimization affects the return value and durable document.saved payload. Full document content remains available through hydration and agent document reads.

Validation pays compilation cost once

Every desktop and utility-process boundary calls parseOrContractError. The old implementation interpreted a schema via Check, then separately created errors for invalid input. The new helper compiles each schema object once and retains the validator in a WeakMap.

const validators = new WeakMap<TSchema, Validator>();

function validatorFor(schema: TSchema): Validator {
  const cached = validators.get(schema);
  if (cached) return cached;
  const validator = Compile(schema);
  validators.set(schema, validator);
  return validator;
}

const validator = validatorFor(schema);
if (validator.Check(value)) return value as Static<Schema>;
const errors = validator.Errors(value);

The same cache is used when recognizing an existing ContractError. Error sanitation and the five-issue limit are unchanged.

Measured path

perf:contracts warms the validator, then validates a 500-block document 100 times. On this run the 234,275-byte fixture measured 0.791 ms median, 2.925 ms p95, and 4.016 ms max. The script fails only when the median exceeds 2 ms.

The cache key is object identity. This works well with the imported schema constants used by the operation registries. Code that constructs a fresh equivalent schema for every call would compile every fresh object.

A burst of activity now does bounded work

The controller already queues activity until an animation frame, with a 100 ms fallback. The old flush reduced over queued items; each item called findIndex over the growing list and copied arrays repeatedly. The replacement builds one ID index, mutates one copied array, then applies the 500-item cap once.

function upsertMany(items: AgentActivity[], activities: AgentActivity[]) {
  const byId = new Map(items.map((item, index) => [item.id, index]));
  const next = [...items];
  for (const activity of activities) {
    const index = byId.get(activity.id);
    if (index === undefined) {
      byId.set(activity.id, next.length);
      next.push(activity);
    } else {
      next[index] = activity;
    }
  }
  return next.slice(-AGENT_ACTIVITY_LIMIT);
}

The behavioral rules remain: an existing ID keeps its position and gets its value replaced; a new ID is appended; only the newest 500 positions survive.

Agent constraint

The system prompt now directs the agent to use blocks as canonical data and plainTextBlocks only for same-revision, single-block text edits. Tables, nested content, multi-block changes, and structural work must become note suggestions.

What future maintainers need to preserve

  • document.save and document.saved must agree on receipt field names across contracts, storage, main, preload, renderer, and fixtures.
  • The receipt's documentRevision must be the concurrency token autosave uses for its next expectedRevision.
  • Changed saves must calculate projectRevision after the project increment and publish the same values returned to the caller.
  • Consumers needing document content must use hydration or an explicit read. Adding blocks back to the receipt would restore serialization and validation cost.
  • Activity state must remain immutable at the array and object level for useMemo and memo to be reliable. The controller currently returns a fresh array and replaces updated objects.
  • Canonical rich blocks cannot be reconstructed from plainTextBlocks. The latter intentionally loses styles, child structure, and non-text content.
Compatibility action required

The durable event shape changed without a version bump. docs/compatibility.md says this is permitted during alpha only, and developers must delete existing application user data before running a build with changed alpha contracts. There is no dual-read path for old document.saved payloads.

Can you carry the model forward?

Open each item after answering it in your own words.

1. After a successful autosave, where does the renderer get the next expected document revision?
From receipt.documentRevision, returned by desktop.saveDocument. The renderer no longer receives a document snapshot from that operation.
2. What would break if a consumer still read event.document.revision?
The new durable event has no document field. Consumers must read the flat event.documentRevision; scope checks use event.projectId and event.documentId.
3. Why does the activity merge create a Map before processing the queued burst?
It resolves an activity ID to its position without scanning the list for every queued item. This retains upsert semantics while changing the merge from repeated searches and copies to one index, one copied array, and one final slice.
4. An agent wants to edit a table cell using an empty plain-text anchor. Which path should it take?
It should create a note suggestion. The canonical table remains in blocks, while plainTextBlocks cannot represent that structure and is restricted to supported single-block text edits.
5. What deployment step follows from changing the durable save event during alpha?
Recreate/delete existing application user data. The policy keeps format and protocol versions at 1 during alpha and provides no migration or dual reader for the old payload.
