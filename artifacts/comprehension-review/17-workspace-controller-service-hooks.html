Workspace controller split into service hooks

This staged change takes the large useWorkspaceController hook and moves most of its side effects into smaller hooks for hydration, document autosave, source import, agent control, and suggestion preview. The main controller now mostly wires those hooks together and keeps the suggestion inbox boundary intact.

The change in one passRead the diff as a separation of responsibilities, with a few behaviour additions around status, cleanup, and guarding duplicate operations.

The controller is now a coordinator

Before this change, useWorkspaceController owned hydration, document persistence, source list reconciliation, agent runtime state, activity upserts, preview insertion, and the desktop subscription in one hook. The current version keeps the public workspace handlers but delegates the stateful parts.

Follow the first render: snapshot fan-out, then ready

Hydration still replaces the editor document from the desktop snapshot when blocks are present. The new part is that the hook also exposes a phase and catches both hydrate failures and initialization failures. The controller passes a stable initializeSnapshot callback into hydration, then fills a ref with the actual fan-out initializer after all child hooks are created.

Invariant to keep: hydration must complete before autosave is allowed to persist editor contents. That is now represented by hydration.phase === "ready", passed into useDocumentAutosave. A future edit that bypasses this gate could save the initial placeholder document over hydrated content.

Autosave pathThis is the most behaviour-heavy extraction. It adds status reporting and a public flush while preserving delayed, serialized document saves.

The document save contract is more explicit

The old autosave logic lived inside the controller. The new hook names the contract: no save before readiness and initialization, debounce editor changes for 650 ms, remove suggestionPreview blocks from persisted content, serialize saves through a promise queue, and advance the expected revision from save results or desktop events.

Two small details matter for later maintenance:

  • .catch(() => undefined) before the next queued save means a previous rejection does not permanently poison the save queue.
  • onDesktopEvent uses Math.max(revisionRef.current, event.document.revision), so an older document.saved event will not move the expected revision backwards.

Tests pinning this down: useDocumentAutosave.test.ts checks debouncing at 650 ms, preview block exclusion, serialized saves with the next revision, suppression before ready, conflict error exposure, and unmount flush.

Desktop eventsThe subscription remains shared, but events are distributed to service hooks before suggestion events are emitted to the feed.

One subscription, narrower handlers

The controller still owns the single desktop.subscribe call. Instead of a large switch, it fans each event into document, source, and agent handlers, then handles suggestion feed events locally. This keeps the desktop boundary in one place while letting each hook decide which event types it cares about.

The new controller test adds a direct listener count check on remount. That is not testing business behaviour; it is guarding against leaked desktop subscriptions as dependencies or child hook callbacks change.

Agent, source, previewThese extractions are smaller, but each carries a behaviour detail that can surprise a maintainer.

Three smaller controllers, three guardrails

Agent commands are now single-flight

useAgentController adds pendingRef. If a start or stop request is already in progress, a second request returns early. The old controller set a pending value but did not have this explicit duplicate guard.

Source imports now expose pending and error state

useSourceController keeps the identity-based reconciliation from the old controller. It also reports pending and error state and suppresses overlapping imports.

Preview cleanup is now owned by the preview hook

Preview insertion is mostly the same as before: do nothing if a preview is already active or the item does not support preview, ignore existing preview blocks when choosing the reference block, insert a suggestionPreview after the last active accepted block, then move the cursor there. The new teardown removes active preview blocks on unmount.

Public boundaryThe controller still gives App.tsx the same core handlers, plus some statuses that are available for future UI.

The return value grew, but callers are not forced to use it

useWorkspaceController still returns inbox, sources, runtime, activity, suggestion navigation, and workspace handlers. New returned fields expose internal state that previously was not visible: workspacePhase, hydrationError, documentSaveStatus, documentSaveError, sourceImportPending, sourceImportError, and flushDocument.

Risks to carry forwardMost risk here is about preserving sequencing and cleanup across hook boundaries.

Maintenance notes for future edits

  • Hydration ordering: useWorkspaceHydration is called before the controller effect that assigns initializeRef.current. With real Promises, hydration completion runs after effects have been registered, which makes this safe. If a future test harness returns a non-compliant thenable that calls synchronously, initialization could be skipped.
  • Autosave readiness: ready must stay tied to hydration.phase === "ready". Saving before that phase risks persisting pre-hydration editor state.
  • Preview teardown: cleanup removes blocks only for the current activePreviewId. That id is kept fresh via a ref. If preview state and editor blocks ever get out of sync, stale preview blocks outside that id would not be removed by this cleanup.
  • UI exposure: save, hydration, and source import errors are returned from the controller but are not yet shown by the current App.tsx path. That is not a bug in this diff, but it affects how visible failures are to users.
  • Test organization: source, agent, and preview tests share workspaceServices.test.ts. That keeps the files compact, but future growth may make failures easier to triage if those hooks get separate test files.

Check your mental model

1. A user edits the document while workspace hydration is still loading. Which code prevents an autosave?

Answer

useDocumentAutosave receives ready from hydration.phase === "ready". Both handleChange and enqueueSave return early when readyRef.current or initializedRef.current is false.

2. What would likely break if suggestionPreview blocks were not filtered before saving?

Answer

Temporary preview content could be persisted as part of the real document. The autosave hook filters editor.document with block.type !== "suggestionPreview" before markdown conversion and desktop.saveDocument.

3. Why does the agent controller keep both React state pending and a ref pendingRef?

Answer

pending exposes UI state. pendingRef gives the async command path an immediate single-flight guard, so a second start or stop call can return before React state updates propagate.

4. If duplicate source rows appear after imports, where would you start debugging?

Answer

Start in useSourceController, especially reconcile and event handling for source.imported. The invariant is that a source is placed first and any existing entry with the same id is removed.

5. Which test best captures the new subscription cleanup concern?

Answer

The new useWorkspaceController.test.ts case that mounts, checks listener count is 1, unmounts, checks 0, remounts, and checks cleanup again. It is tied to the shared desktop.subscribe effect in the controller.
