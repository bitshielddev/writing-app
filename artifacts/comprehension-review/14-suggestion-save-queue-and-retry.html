Comprehension review / suggestion persistence

A single-flight save path for suggestion state

This change moves suggestion snapshot writes out of the workspace controller and into a focused hook that sequences saves, coalesces pending projections, preserves failed work for retry, and shows the writer when local suggestion changes might not survive restart.

The old path treated dispatch as if it were durability

Before this diff, useWorkspaceController handed every inbox projection directly to desktop.saveSuggestionState. The call was fire-and-forget: errors were logged, but the UI kept operating with no visible indication that storage had rejected the latest suggestion state.

The task file for this change, .tasks/02-sequence-suggestion-saves-and-surface-failures.md, names two constraints that shape the implementation: keep the bridge method unchanged for now, and do not hide restart-loss risk after a failed save. This is explicitly an interim renderer-side correctness step before storage becomes the authoritative writer in Task 10.

Behavioral change to remember: a suggestion reducer update is no longer equivalent to a durable write. The renderer tracks which local projection has actually been acknowledged by the desktop bridge.

The queue has one active slot and one replaceable waiting slot

The new hook models persistence as complete projections, not field patches. Every reducer output is a full PersistedSuggestionState, so while one save is active the newest waiting snapshot can replace the older waiting snapshot. That avoids resurrecting removed entries through a partial merge.

1. Inbox changesuseSuggestionInbox emits a persisted projection through onStateChange.
2. requestSaveThe hook ignores pre-hydration writes, assigns a renderer-local version, and clones the state.
3. pending slotThe candidate becomes pendingRef. If another save is active, no second bridge call starts yet.
4. drainWhen free, pending moves to active and desktop.saveSuggestionState is called.
5. ack or failSuccess advances acknowledgedVersion. Failure retains the newest unacknowledged candidate for retry.
The public status type is intentionally small. The versions are local bookkeeping, not database revisions.
export type SuggestionPersistenceStatus =
  | { state: "idle"; acknowledgedVersion: number }
  | {
      state: "saving" | "pending" | "retrying";
      acknowledgedVersion: number;
    }
  | { state: "failed"; acknowledgedVersion: number; message: string };
A save request is copied and coalesced before the bridge sees it.
const requestSave = useCallback(
  (state: PersistedSuggestionState) => {
    if (!hydratedRef.current || !mountedRef.current) return;
    if (hydratedEchoRef.current) {
      const isHydrationEcho =
        JSON.stringify(state) === JSON.stringify(hydratedEchoRef.current);
      hydratedEchoRef.current = undefined;
      if (isHydrationEcho) return;
    }
    const candidate = {
      version: ++nextVersionRef.current,
      state: immutableCopy(state),
    };
    latestProjectionRef.current = candidate.state;
    pendingRef.current = candidate;

    if (activeRef.current) {
      publishStatus({
        state: "pending",
        acknowledgedVersion: acknowledgedVersionRef.current,
      });
      return;
    }

    failedRef.current = undefined;
    drain();
  },
  [drain, publishStatus],
);

What to notice in the save request path

Line of behaviorWhy it matters
!hydratedRef.current returns earlySuggestion persistence remains disabled until the desktop snapshot has seeded the hook.
JSON.stringify compares the first hydrated echoThe reducer's hydrate dispatch can produce an onStateChange event. If it matches the seeded snapshot, it is not written back to storage.
immutableCopy(state)The bridge receives a snapshot that later local mutations cannot alter. In this implementation the copy is structuredClone.
pendingRef.current = candidateWhile active work is running, later projections replace the one waiting slot. The newest complete projection wins.
failedRef.current = undefined before draining a new requestA later user action supersedes the failed retry target and starts persistence again through the same queue.
The drain function is the single place that starts bridge calls.
const drain = useCallback(function drainQueue() {
  if (!mountedRef.current || activeRef.current || !pendingRef.current) return;

  const candidate = pendingRef.current;
  const retrying = retryingRef.current;
  pendingRef.current = undefined;
  activeRef.current = candidate;
  retryingRef.current = false;
  publishStatus({
    state: retrying ? "retrying" : "saving",
    acknowledgedVersion: acknowledgedVersionRef.current,
  });

  void desktop.saveSuggestionState(candidate.state).then(
    () => {
      activeRef.current = undefined;
      acknowledgedVersionRef.current = candidate.version;
      failedRef.current = undefined;
      console.info(`Suggestion state save acknowledged (version ${candidate.version})`);

      if (!mountedRef.current) return;
      if (pendingRef.current) {
        publishStatus({
          state: "pending",
          acknowledgedVersion: acknowledgedVersionRef.current,
        });
        drainQueue();
        return;
      }

      setFailureMessage(undefined);
      publishStatus({ state: "idle", acknowledgedVersion: acknowledgedVersionRef.current });
    },
    (error: unknown) => {
      activeRef.current = undefined;
      failedRef.current = pendingRef.current ?? candidate;
      pendingRef.current = undefined;
      const detail = error instanceof Error ? error.message : String(error);
      console.error(
        `Suggestion state save failed (version ${candidate.version}): ${detail}`,
      );
      if (!mountedRef.current) return;
      setFailureMessage(SAVE_FAILURE_MESSAGE);
      publishStatus({
        state: "failed",
        acknowledgedVersion: acknowledgedVersionRef.current,
        message: detail,
      });
    },
  );
}, [desktop, publishStatus]);

Hydration is now seeded before the inbox is hydrated

The controller calls seedHydratedState(snapshot.suggestions) immediately before hydrateInbox(snapshot.suggestions). That order is the key to suppressing the hydration echo. The hook knows what storage already returned before the reducer installs it and before useSuggestionInbox can publish the resulting projection.

Failure now stays visible until an acknowledged save clears it

The hook separates the internal failure detail from the user-facing warning. status.message carries the bridge error, such as disk full in the test. failureMessage is the concise UI text: Suggestion changes may not survive an application restart.

A retry is allowed only when there is no active save, no pending save, and a retained failed candidate exists. A later reducer change can also restart persistence; in that case it replaces the failed candidate with the newest local projection.

Maintenance risk: the alert remains present during a new save attempt after failure because failureMessage is cleared only after success. The Retry button is disabled unless status.state === "failed", so the UI can warn without allowing a duplicate retry while saving.

Edges worth carrying in your head

  • acknowledgedVersion is renderer-local. Do not treat it as a database revision or cross-process concurrency token.
  • The hydration echo check compares JSON strings. That is acceptable here because the state is expected to be JSON-safe persisted suggestion data, but key-order assumptions would matter if this shape became less plain.
  • Only the newest pending projection is retained. This is deliberate because each projection is complete state.
  • On failure, the retained retry target is pendingRef.current ?? candidate. If local changes arrived while the failed save was active, retry sends those newer changes.
  • The hook logs version and error context but does not log suggestion bodies.

Pause and check the model

1. A save for version 1 is active. Versions 2 through 11 are requested before version 1 resolves. How many bridge calls should have started before version 1 resolves?

One. The active version 1 call remains the only in-flight call. Versions 2 through 11 replace the pending slot, so the next bridge call starts only after version 1 resolves, and it sends version 11.

2. Why does the controller call seedHydratedState before hydrateInbox?

The hook needs to know the exact persisted suggestion snapshot before the inbox reducer installs it. When the reducer publishes its hydrated state, requestSave can recognize it as the hydration echo and skip writing it back.

3. What would break if pending snapshots were merged field-by-field instead of replaced?

Old data could be resurrected. The reducer emits complete projections; replacing the pending candidate preserves the latest whole state. Field merging could combine newer fields with stale entries or pins that the reducer had removed.

4. After a save fails while a newer pending projection exists, which projection does Retry send?

The newer pending projection. The failure handler assigns failedRef.current = pendingRef.current ?? candidate, then clears pending. Retry moves that retained candidate back into pending and drains it.

5. Where would you start debugging if users see the warning after clicking Retry?

Check useSuggestionPersistence.retry and drain: whether failedRef.current exists, whether an active or pending save blocks retry, and whether the bridge promise resolves or rejects. The dock only displays failureMessage and calls the provided retry callback.
