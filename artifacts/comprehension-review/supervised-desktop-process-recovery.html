Supervised utility processes, visible health, and safer shutdown

This change turns storage and agent utility processes into monitored runtime dependencies. The main process now tracks their health, retries bounded failures, exposes that state to the renderer, and gives tests a way to kill the child processes and verify recovery.

Working tree: 27 changed files; outside package-lock.json the staged diff is about 746 insertions and 57 deletions.
Most affected path: Electron main process, child RPC, renderer health UI, and E2E test harness.
Validation run for this review: npm test -- --run desktop/process-supervisor.test.ts desktop/child-rpc.test.ts passed with 15 tests.

The runtime now has a health loop

The main behavioural change is a loop around each utility process. Before this diff, desktop/main.ts spawned storage and agent processes during startup and killed them during shutdown or document switching. The new path adds a supervisor that can spawn, validate, recover, mark health, and stop after a bounded number of retries.

spawn
Create storage or agent utility process through spawnChild.
validate
Call health.ping over child RPC.
recover
Repair/hydrate storage, or seed agent state from storage.
publish
Broadcast process.health and unblock UI paths only when healthy.

The new supervisor is small enough to read directly. Its key invariant is that automatic restart attempts are bounded by RESTART_DELAYS_MS, and a non-retryable error becomes a stable failed state.

export const RESTART_DELAYS_MS = [500, 2_000, 5_000] as const;
export const RESTART_WINDOW_MS = 5 * 60_000;

export class ProcessSupervisor<Process extends SupervisedProcess> {
  private process?: Process;
  private attempts: number[] = [];
  private cycle?: Promise<Process>;
  private currentHealth: ProcessHealth = { state: "starting" };

  start() {
    this.stopped = false;
    return this.beginCycle(false);
  }

  processFailed(error: unknown) {
    if (this.stopped) return Promise.reject(error);
    this.publish({ state: "degraded", reason: errorMessage(error) });
    return this.beginCycle(false, error);
  }

  retry() {
    this.attempts = [];
    this.healthySince = undefined;
    this.stopped = false;
    return this.beginCycle(true);
  }
}
What to notice: processFailed does not spawn directly. It enters the same serialized cycle as startup and manual retry. That cycle promise prevents overlapping restarts from racing each other.

The restart decision is made in prepareAttempt. It filters old attempt timestamps, stops immediately on non-retryable errors, and stops after the configured retry count.

private prepareAttempt(error: unknown, manual: boolean) {
  const now = this.options.now?.() ?? Date.now();
  if (this.healthySince !== undefined && now - this.healthySince >= HEALTHY_RESET_MS) this.attempts = [];
  this.attempts = this.attempts.filter((time) => now - time < RESTART_WINDOW_MS);
  if (error && this.options.classifyRetryable?.(error) === false) {
    this.publish({ state: "failed", reason: errorMessage(error) });
    throw error;
  }
  if (this.attempts.length >= RESTART_DELAYS_MS.length) {
    const reason = error ? errorMessage(error) : "Automatic restart limit reached";
    this.publish({ state: "failed", reason });
    throw error instanceof Error ? error : new Error(reason);
  }
  return { restarting: Boolean(error || manual), attempt: this.attempts.length + 1, now };
}

Where the process state crosses boundaries

The diff adds health as a contract, not just a local main-process detail. That matters because the renderer uses the state to change editing behaviour and offer manual retry.

The user-facing behaviour is deliberately asymmetric

Storage health and agent health do not affect the UI in the same way. The change treats storage as authoritative for local workspace mutations, while the agent is optional for basic editing and saving.

ConditionRenderer behaviourMain-process guard
Storage is not healthy Show a red alert, make the editor read-only, and stop enqueueing autosaves. Reject storage mutations with STORAGE_UNAVAILABLE.
Agent is not healthy Show an amber status banner. Editing and saving continue. Reject agent.start and agent.stop with AGENT_UNAVAILABLE. 
function ProcessHealthBanner({ health, canRetry, retry }: { ... }) {
  if (health.storage.state !== "healthy") {
    const retryable = health.storage.state === "failed" || health.storage.state === "degraded";
    return (
      <div role="alert" className="... bg-red-800 ...">
        <span>Storage is unavailable. Editing and workspace changes are read-only; unsaved text is retained.</span>
        {retryable && canRetry ? <button onClick={() => void retry("storage")}>Retry storage</button> : null}
      </div>
    );
  }
  if (health.agent.state === "healthy") return null;
  // agent banner leaves document editing available
}

The autosave hook gets the same storage-health gate. It does not enqueue a save when storage is unhealthy, and if storage becomes healthy after a failed save, it attempts another save.

const enqueueSave = useCallback(() => {

Maintenance note: the banner text says unsaved text is retained. The code supports that by avoiding destructive writes while storage is down and by retrying a failed save when storage recovers. It does not make the child operation itself durable during a crash; the retention here is renderer/editor state plus retry after failure.

RPC calls now have deadlines and cancellation messages

The supervisor depends on health.ping, startup deadlines, and predictable shutdown. Those behaviours are built into ChildRpc in this diff.

Deadlines

Read calls default to 10s, mutations to 15s, agent control to 30s, startup to 60s, and shutdown to 10s.

Cancellation

A timed-out or aborted call posts rpc.cancel and clears its pending entry.

Graceful shutdown

shutdown() posts a shutdown message, waits for disposal, and kills the child only after the grace period.

async callWithOptions<Name extends OperationName<Registry>>(
  operation: Name,
  options: RpcCallOptions,
  ...args: OperationArgs<Registry, Name>
): Promise<OperationResult<Registry, Name>> {
  await this.ready;
  const deadlineMs = options.deadlineMs ?? deadlineFor(operation);
  return new Promise((resolve, reject) => {
    const finishFailure = (error: Error) => {
      const request = this.pending.get(id);
      if (!request) return;
      this.pending.delete(id);
      clearTimeout(request.timer);
      this.child.postMessage({
        kind: "rpc.cancel",
        protocolVersion: PROTOCOL_VERSION,
        id,
        operation,
      });
      reject(error);
    };
    const timer = setTimeout(
      () => finishFailure(new OperationTimeoutError(operation, deadlineMs)),
      deadlineMs,
    );
    const abort = () => finishFailure(new OperationCancelledError(operation));
    // pending registration and postMessage follow
  });
}

The child sides understand cancellation, but the semantics are narrow: cancellation suppresses a late response after work finishes. The changed code does not interrupt a storage operation mid-query or abort an agent start mid-call.

Main-process recovery is scoped differently for storage and agent

The two supervisors use the same lifecycle mechanics, but their recovery steps are different because storage owns workspace state and the agent owns per-document runtime work.

Storage recovery

const storageSupervisor = new ProcessSupervisor({
  spawn: () => storage = spawnChild(...),
  validate: async (process) => { await process.call("health.ping"); },
  recover: async (process) => {
    const catalog = await process.call("workspace.catalog");
    selectedScope = catalog.selection;
    await process.call("workspace.repair", selectedScope);
    await process.call("hydrate", selectedScope);
  },
  classifyRetryable: retryable,
  onHealth: (next) => setProcessHealth("storage", next),
});

Evidence from the code: storage recovery reads the catalog, restores the selected scope, repairs the selected workspace, and hydrates it before reporting healthy.

Agent recovery

recover: async (process) => {
  const seed = await storage.call("agent.seed", scope);
  process.post({
    kind: "project.changed", protocolVersion: PROTOCOL_VERSION,
    streamId: seed.streamId, sequence: seed.coveredThroughSequence,
    projectRevision: seed.projectRevision,
    documentRevision: seed.documentRevision,
  });
  if (agentShouldRun) {
    await process.call("agent.start", {
      ...scope,
      projectRevision: seed.projectRevision,
      documentRevision: seed.documentRevision,
    });
  }
}

Evidence from the code: agent recovery reseeds the agent from storage and restarts the agent only if the remembered runtime state says it should be running.

The retry classifier is shared. The inferred intent is to avoid retry loops for compatibility, migration, or corruption errors, because those are unlikely to fix themselves by respawning the process.

const retryable = (error: unknown) => !(error instanceof Error && (
  error.message.includes("compatibility mismatch") ||
  ("code" in error && typeof error.code === "string" && (
    error.code.includes("PROTOCOL") || error.code.includes("UNSUPPORTED") ||
    error.code.includes("MIGRATION") || error.code.includes("DATABASE_CORRUPT")
  ))
));

Shutdown now flushes renderer state before exiting

The previous before-quit handler posted shutdown to the agent and killed both children. The new path prevents the quit, asks renderer windows to flush, asks both children to shut down, then exits.

app.on("before-quit", (event) => {
  if (quitting) return;
  event.preventDefault();
  quitting = true;
  void (async () => {
    await Promise.allSettled(BrowserWindow.getAllWindows().map((window) =>
      window.webContents.executeJavaScript("window.scribeFlush?.()")));
    await Promise.allSettled([agent?.shutdown(), storage?.shutdown()]);
    app.exit(0);
  })();
});

The renderer side wires that global to a workspace flush that covers both document content and suggestion persistence.

useEffect(() => {
  window.scribeFlush = flushDocument;
  return () => { delete window.scribeFlush; };
}, [flushDocument]);

const flushWorkspace = useCallback(async () => {
  await Promise.allSettled([document.flushForSwitch(), suggestionPersistence.flush()]);
}, [document, suggestionPersistence]);
Invariant to keep: anything that must be persisted on app quit needs to be reachable from window.scribeFlush. This change currently includes document saves and suggestion persistence. If another local buffer is added later, it should join flushWorkspace or an equivalent quit-flush path.

Risks and assumptions worth keeping in your head

Cancellation is not execution abort. The parent stops waiting and the child suppresses a late response, but the underlying storage or agent work can still run to completion. Do not assume rpc.cancel rolls back a mutation.
Health gates are broad. ipc-routing.ts blocks a fixed set of storage mutation operation names while storage is not healthy. If a new mutation is added later, it must be added to that set or covered by a more central classification.
Agent restart depends on agentShouldRun. That flag is inferred from runtime status changes in setRuntime. If future statuses are added, the flag logic may need to change so recovery restarts only when intended.
The E2E-only bridge is gated by SCRIBE_E2E. This is supported by secureWebPreferences adding --scribe-e2e only when the environment variable is set, and preload exposing window.scribeTest only with that argument. Keep that gate intact when changing test controls.

Comprehension check

1. If storage is killed during an editing session, which layers react before the user can edit again?

Answer: The storage ChildRpc disposes on exit, the storage supervisor publishes degraded or restarting, desktop/main.ts broadcasts process.health, useWorkspaceController stores the new health snapshot, and App renders the storage alert while passing editable={false} to the editor. Editing becomes available again after the supervisor validates and recovers storage and publishes healthy.

2. What would break if a new storage mutation operation were added only to RendererOperations?

Answer: It might bypass the unhealthy-storage guard in ipc-routing.ts, because the guard uses a hand-maintained storageMutations set. The operation would still need contract schemas and routing, but the health safety rule would be incomplete.

3. Does rpc.cancel stop an in-flight database mutation?

Answer: No evidence in the changed code shows mid-operation abort. Storage records the canceled id, awaits handleRequest, then suppresses success or failure if the id was canceled. The parent pending entry is cleaned up, but the child-side work can finish.

4. Why does agent recovery call storage.call("agent.seed", scope) before optionally restarting the agent?

Answer: The recovered agent process needs current stream and revision data for the selected document. The code posts project.changed with the seed revisions, then calls agent.start only when agentShouldRun is true.

5. Where would you start debugging if the packaged app opens but the CI smoke test times out?

Answer: Start with scripts/run-packaged-smoke.mjs and desktop/main.ts's SCRIBE_E2E_READY log. The script waits for that exact stdout/stderr text, and main only logs it after the workspace window fires did-finish-load when SCRIBE_E2E=1.
