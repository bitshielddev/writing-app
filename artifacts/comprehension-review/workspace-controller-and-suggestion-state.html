Where orchestration moved, and where suggestion state now lives

This working tree turns a crowded renderer and dock into smaller boundaries, separates agent runtime from suggestion events, and shares the persisted suggestion policy between React and Electron storage. The behavior is mostly a preservation refactor, with a few contract changes that future work needs to respect.

The new ownership map

The important shift is ownership, not a new user-facing feature. `App.tsx` now composes layout and delegates the stateful desktop/editor work to `useWorkspaceController`. Suggestion state types have moved out of the desktop bridge contract and into `src/suggestions/state.ts`, so renderer and storage use the same limit and empty projection.

Follow a document change through the new controller

The old `App.tsx` held the feed, inbox, hydration, save queue, source list, runtime, agent controls, and preview handlers inline. That logic is now in `src/workspace/useWorkspaceController.ts`. `App.tsx` still wires the returned handlers to UI, but it no longer knows how to save a document or start the agent.

`src/App.tsx`: layout consumes one controller object instead of owning the orchestration itself.

Autosave behavior is preserved but now easier to test directly. The controller waits until hydration has completed, removes `suggestionPreview` blocks from the saved document, serializes saves through `saveQueueRef`, and advances the expected document revision after each successful desktop save.

`src/workspace/useWorkspaceController.ts`: the save queue keeps revision expectations ordered.
const persistDocument = useCallback(() => {
  if (!documentHydratedRef.current || hydrationInProgressRef.current) return;
  const blocks = editor.document.filter(
    (block) => block.type !== "suggestionPreview",
  );
  const markdown = editor.blocksToMarkdownLossy(blocks);
  saveQueueRef.current = saveQueueRef.current
    .then(async () => {
      const document = await desktop.saveDocument({
        documentId: documentIdRef.current,
        blocks,
        markdown,
        expectedRevision: documentRevisionRef.current,
      });
      documentRevisionRef.current = document.revision;
    })
    .catch((error: unknown) => {
      console.error("Document save failed", error);
    });
}, [desktop, editor]);

Preview insertion follows the same path as before: only text suggestions can preview, only one preview may be active, and the insertion target is the last active accepted block or the final accepted block. The type guard now comes from `suggestions/types.ts`, not a local helper.

The durable suggestion projection is now a shared module

`PersistedSuggestionState` used to be declared in `src/shared/desktop.ts`. This diff moves it into `src/suggestions/state.ts` and re-exports the type from the desktop contract. That is more than tidying imports: storage and the reducer now call the same empty-state factory and trimming function.

`src/suggestions/state.ts`: one definition for the persisted shape and 30-entry policy.
export const SUGGESTION_ENTRY_LIMIT = 30;

export type PersistedSuggestionState = {
  entries: PersistedInboxEntry[];
  pinnedEntries: PersistedPinnedEntry[];
  workspacePins: PersistedWorkspacePin[];
  seenKeys: Record<string, true>;
  nextZIndex: number;
};

export function trimSuggestionEntries<T extends PersistedInboxEntry>(
  entries: readonly T[],
  protectedIds: Iterable<string> = [],
): T[] {
  if (entries.length <= SUGGESTION_ENTRY_LIMIT) return [...entries];

  const protectedSet = new Set(protectedIds);
  const evictionOrder = entries
    .filter((entry) => !protectedSet.has(entry.item.id))
    .sort((left, right) => {
      if (left.viewed !== right.viewed) return left.viewed ? -1 : 1;
      return left.item.createdAt - right.item.createdAt;
    });
  ...
}

The reducer uses the same trim policy, but adds renderer-only protection for the currently selected item and active preview. That preserves the old invariant: adding a new suggestion should not evict the item the user is reading or previewing.

`src/suggestions/inbox.ts`: persisted state is the base; selection and preview stay ephemeral.
export type InboxState = PersistedSuggestionState & {
  selectedId?: string;
  activePreviewId?: string;
};

function protectedQueueIds(state: InboxState): string[] {
  return [state.selectedId, state.activePreviewId].filter(
    (id): id is string => id !== undefined,
  );
}

function trimQueue(state: InboxState): InboxState {
  const entries = trimSuggestionEntries(state.entries, protectedQueueIds(state));
  return entries.length === state.entries.length ? state : { ...state, entries };
}

The other contract change is that agent status no longer enters the inbox reducer. `SuggestionEvent` now contains only `suggestion.added`, `suggestion.updated`, and `suggestion.retracted`. Status and errors arrive as `AgentRuntime` through `DesktopEvent` and are stored by the workspace controller.

`src/desktop/desktopClient.ts`: the feed adapter ignores runtime events now.
export function createDesktopSuggestionFeed(
  bridge: DesktopBridge,
): SuggestionFeed {
  return {
    subscribe(listener) {
      return bridge.subscribe((event) => {
        if (event.type === "suggestion.event") listener(event.event);
      });
    },
  };
}

Agent activity and mutation safety moved behind small helpers

`desktop/agent-events.ts` extracts session-event-to-activity conversion from `desktop/agent.ts`. The conversion keeps stable identities for tool calls and splits message text, reasoning, and provider errors into separate activities. Electron main still stores activity in the in-memory ring and broadcasts `agent.activity` events.

`desktop/agent-events.ts`: session events become renderer activity records.
export function activitiesFromSessionEvent(
  event: AgentSessionEvent,
  now = Date.now(),
): ActivityInput[] {
  switch (event.type) {
    case "agent_start":
    case "agent_end":
      return observeLifecycle(event, now);
    case "message_start":
    case "message_update":
    case "message_end":
      return observeMessage(event);
    case "tool_execution_start":
    case "tool_execution_update":
    case "tool_execution_end":
      return observeToolActivity(event, now);
    default:
      return [];
  }
}

The Pi extension now has one mutation helper for create, update, and retract. It refuses to mutate suggestions when no active document revision exists, passes `expectedDocumentRevision` to storage, and wakes the loop if storage rejects the mutation. The storage side enforces that revision with `assertCurrentRevision`, so stale suggestions do not get committed against a newer draft.

`desktop/scribe-extension.ts`: all suggestion mutations pass through the active revision check.
export async function executeSuggestionMutation(
  host: ScribeExtensionHost,
  method: string,
  params: Record<string, unknown>,
) {
  const expectedDocumentRevision = host.loop.snapshot().activeDocumentRevision;
  if (expectedDocumentRevision === undefined) {
    return toolResult("No active revision", true);
  }
  try {
    return toolResult(
      await host.storageCall(method, { ...params, expectedDocumentRevision }),
    );
  } catch (error) {
    host.wake();
    return toolResult(
      error instanceof Error ? error.message : String(error),
      true,
    );
  }
}

Electron main also gets two small hardening changes: IPC registration now goes through `registerValidatedIpc`, and both workspace and mock suggestion windows use `secureWebPreferences`, which includes `sandbox: true` alongside the existing context isolation and disabled node integration.

The dock is now four files with one public component

`SuggestionDock.tsx` used to contain queue rows, source labels, detail view, activity view, toolbar, and routing logic in one file. The public props are still consumed by `SuggestionDock`, but rendering is delegated to `SuggestionDockActivity`, `SuggestionDockDetail`, and `SuggestionDockQueue`.

Kind-family decisions are also centralized. `SuggestionDockDetail`, `WorkspacePins`, `workspacePinLayout`, mock suggestions, validation, and keybindings now call guards such as `isTextSuggestion`, `isStructureSuggestion`, and `isVisualSuggestion`. This matters if a future suggestion kind is added, because the list of places to update is smaller and documented.

Check your mental model

1. If a component wants to display "Agent stopped" or "Agent error", should it subscribe to `SuggestionFeed`?

No. Runtime status and errors now come from `AgentRuntime` through `DesktopEvent` and the workspace controller. `SuggestionFeed` emits only committed suggestion events.

2. Why does the reducer pass protected ids into `trimSuggestionEntries`?

Selected and active-preview suggestions must not be evicted while the user is reading or editing them. The shared trim function supports protected ids so the renderer can preserve those ephemeral states.

3. What would break if `App.tsx` started constructing a new desktop suggestion feed every render?

The inbox subscription could be torn down and recreated repeatedly, which risks duplicated or missed event handling under React rendering and StrictMode. The controller memoizes the feed to keep the subscription boundary stable.

4. Where would you start debugging if suggestion create/update tools started failing with `STALE_SUGGESTION_REVISION`?

Start at `executeSuggestionMutation` and storage's `assertCurrentRevision`. The helper sends `expectedDocumentRevision` from the active loop snapshot; storage compares it with the current document revision. A mismatch means the agent is acting on an old draft revision and should be woken to review the latest one.

5. When adding a new suggestion kind, what is the safe first place to change?

Start in `src/suggestions/types.ts`: add the literal to `SUGGESTION_KINDS`, define its union member, and place it in the appropriate kind-family guard or add a new guard. Then update validation, presentation, preview/workspace behavior, mock tooling, and tests.
