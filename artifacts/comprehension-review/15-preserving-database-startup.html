Database startup now preserves first and asks for help second

This change moves SQLite opening out of desktop/storage.ts into a dedicated database lifecycle module. The behaviour that matters most: existing workspace data is no longer dropped just because user_version is unexpected.

Working tree review Targeted tests: 22 passed Built on recent desktop boundary work

The startup path affected by this diff

The previous startup path opened the storage database inside desktop/storage.ts, checked PRAGMA user_version, and rebuilt the schema if the version was not exactly 2. The new path inspects the database before opening it for writes, classifies failure modes, and sends a structured startup error back to the Electron main process.

1
Main starts storage first.
startDesktop() still waits for storage.ready before spawning the agent or creating a window.
2
Storage calls openApplicationDatabase(dbPath).
The inspection and schema creation/migration rules now live in desktop/database.ts.
3
Known database failures become startup.error.
The child process posts a structured message before throwing, so the parent can distinguish database compatibility problems from a generic utility-process crash.
4
Main shows recovery guidance and quits.
Only ChildStartupErrors whose code starts with DATABASE_ get the database-specific dialog text.

History context. The recent a3def0f / PR #2 desktop boundary work introduced the ChildRpc, startup orchestration, and process contracts this diff extends. The latest committed feature, f2a6258, focused on suggestion persistence. This change returns to the desktop storage boundary and hardens startup before more state accumulates in SQLite.

The behavioural turn: from reset-on-mismatch to classify-before-write

The most consequential removed code was the unconditional reset when user_version differed from the compiled database version. That treated older, newer, and partly corrupted databases the same way.

The new invariant is visible in the catch block: if storage cannot safely open the database, it reports the failure and then stops. It does not fall back to an empty workspace.

How the new opener decides what is safe

openApplicationDatabase() begins with a read-only inspection when the database file already exists. That order matters because newer or suspicious files should not be opened for writes before the app knows what they are.

export function openApplicationDatabase(
  databasePath: string,
  migrations: readonly DatabaseMigration[] = [],
) {
  validateMigrationRegistry(migrations);
  let initialInspection: DatabaseInspection = { kind: "empty", version: 0 };
  if (databasePath !== ":memory:" && existsSync(databasePath)) {
    let inspectionDb: DatabaseSync | undefined;
    try {
      inspectionDb = openReadOnly(databasePath);
      initialInspection = inspectDatabase(inspectionDb);
    } catch (error) {
      throw new DatabaseStartupError(
        "DATABASE_CORRUPT",
        databasePath,
        error instanceof Error ? error.message : String(error),
        { cause: error },
      );
    } finally {
      inspectionDb?.close();
    }
  }
  const inspectionError = errorForInspection(initialInspection, databasePath);
  if (inspectionError) throw inspectionError;

  const db = new DatabaseSync(databasePath, {
    enableForeignKeyConstraints: true,
    timeout: 5_000,
  });
  try {
    db.exec("PRAGMA journal_mode = WAL");
    db.exec("PRAGMA synchronous = FULL");
    if (initialInspection.kind === "empty") createCurrentDatabase(db);
    if (initialInspection.kind === "older") {
      runMigrations({ db, databasePath, migrations });
    }
    const finalInspection = inspectDatabase(db);
    const finalError = errorForInspection(finalInspection, databasePath);
    if (finalError) throw finalError;
    return db;
  } catch (error) {
    db.close();
    throw error;
  }
}

The migration framework is stricter than the current schema needs

The code now defines the rules future migrations must obey. A migration must advance exactly one version, the registry must form the full contiguous path, and risky migrations can ask for a checkpointed backup before running.

function migrationPath(
  migrations: readonly DatabaseMigration[],
  fromVersion: number,
  targetVersion: number,
) {
  const edges = new Map<number, DatabaseMigration>();
  for (const migration of migrations) {
    if (migration.toVersion !== migration.fromVersion + 1) {
      throw new Error(`Migration ${migration.name} must advance exactly one version`);
    }
    if (edges.has(migration.fromVersion)) {
      throw new Error(`Duplicate migration from version ${migration.fromVersion}`);
    }
    edges.set(migration.fromVersion, migration);
  }
  const path: DatabaseMigration[] = [];
  for (let version = fromVersion; version < targetVersion; version += 1) {
    const migration = edges.get(version);
    if (!migration || migration.toVersion !== version + 1) {
      throw new Error(`No contiguous migration path from version ${version} to ${targetVersion}`);
    }
    path.push(migration);
  }
  return path;
}

When a risky migration fails, the schema transaction rolls back and the backup name is changed from .bak to .failed.bak. The failed backup is intentionally excluded from successful-backup pruning.

What is protected

Versioned fixture rows for projects, documents, sources, suggestion state, and outbox are asserted to survive a normal open. Failed migration tests assert the partial table is gone after rollback and the original fixture row remains.

What remains a future concern

No real migration is registered by this change. The tests use synthetic migrations through runMigrations(), so the next schema release will still need to wire its migration into openApplicationDatabase().

The error now survives the process boundary

The shared contract adds a specific child message for startup failures. ChildRpc validates the shape before trusting it, then rejects ready with a typed error.

The user-facing failure is intentionally narrow

The Electron dialog is only shown when the error reaches main as a ChildStartupError with a database-prefixed code. A crashed storage process still follows the existing generic Desktop startup failed logging and quit path.

export function databaseStartupGuidance(error: unknown) {
  if (!(error instanceof ChildStartupError) || !error.code.startsWith("DATABASE_")) {
    return undefined;
  }
  const path = error.databasePath ?? "the local workspace database";
  return [
    "ScribeAI could not open the writing workspace.",
    error.message,
    `Your data was not reset. Preserve ${path} and any adjacent .bak files before attempting recovery.`,
  ].join("\n\n");
}
void runDesktopStartup(
  async () => {
    await app.whenReady();
    await start();
  },
  (error) => {
    console.error("Desktop startup failed", error);
    const guidance = databaseStartupGuidance(error);
    if (guidance) dialog.showErrorBox("Workspace database could not be opened", guidance);
    app.quit();
  },
);

Debugging clue: if a user reports seeing the new dialog, the failure happened before the workspace window was created and before the agent process was spawned. Start with scribe.sqlite3, adjacent .bak files, and the exact DATABASE_* code in the message.

Maintenance notes and edge cases

Schema drift risk

CURRENT_SCHEMA_SQL, APPLICATION_TABLES, the schema fixture, and storage's query expectations must stay aligned. If a table is added but omitted from APPLICATION_TABLES, inspection will not verify it.

Migration registration risk

openApplicationDatabase() defaults to an empty migration list. When DATABASE_VERSION increases, the caller or module will need to provide the migration registry or validation will reject older supported files.

Filesystem assumptions

Risky backups use wal_checkpoint(TRUNCATE), copy the database file, fsync the copy, and validate it read-only. This is stronger than a raw copy, but recovery still depends on preserving files beside scribe.sqlite3.

Message identity

databaseStartupGuidance() relies on instanceof ChildStartupError. That is safe for the current in-process main-side error object from ChildRpc; a plain serialized error would not trigger guidance.

Comprehension check

Use these as a quick self-check before modifying this path.

1. A user's database has PRAGMA user_version = 99. What path does startup take now?

inspectDatabase() returns { kind: "newer", version: 99 }. errorForInspection() turns that into DATABASE_TOO_NEW. Storage posts startup.error, ChildRpc.ready rejects with ChildStartupError, main shows database recovery guidance, and the app quits without rewriting the file.

2. Why does the code inspect an existing file with a read-only connection before opening the write connection?

It avoids writing to files that are newer, legacy-unknown, corrupt, foreign-key-invalid, or missing required tables. The tests for newer and legacy-unknown databases assert the file bytes are unchanged after rejection.

3. What would break if malformed startup.error messages were accepted without validation?

A child process could reject startup with a poorly shaped error, such as a numeric code or non-string path. That would weaken the process contract and could produce misleading guidance. The new test confirms malformed startup errors are ignored and a later ready can still resolve startup.

4. A risky migration creates a table and then throws. Which invariants should hold afterward?

The transaction should roll back, so the partial table should not exist. user_version should remain at the original version. Existing data should still be readable. The backup should be retained with a .failed.bak suffix and validate as a supported database.

5. Where would you start debugging if the new dialog appears before any window opens?

Start in the storage startup path: inspect the reported DATABASE_* code, the database path in the guidance, and any adjacent migration backups. The UI window and agent are downstream of storage.ready, so they are unlikely to be the first cause.
