Exact-text edits replace inserted previews

This staged change turns suggestions from broad content cards into a smaller contract: an agent can propose an edit anchored to current draft text, a note, or a Mermaid diagram. The main behavior to understand is the new edit path: preview focuses the source, accept re-checks the source, then replaces it.

Read the diff as a boundary change first

The domain union in src/domain/suggestions/schema.ts is the highest-leverage part of this change. Legacy kinds such as snippet, fact, term, outline, layout, and mindMap no longer validate as current suggestions. The replacement set is much smaller.

Why this matters

The old preview model inserted a temporary editable block. The new model treats the suggestion as a proposed replacement for exact draft text. That changes where correctness lives: the renderer must prove the source text is still present and unique at the moment of preview or acceptance.

1. Matching is exact and document-wide

getEditSuggestionStatus walks accepted editor blocks, extracts visible text from string or nested BlockNote-ish content, and counts exact sourceText occurrences. The edit is enabled only if there is one match.

export function getEditSuggestionStatus(
  editor: WritingEditor,
  item: EditSuggestion,
): EditSuggestionStatus {
  const matches: Array<{ block: WritingBlock; start: number; end: number }> = [];
  for (const block of editor.document) {
    if (block.type === "suggestionPreview") continue;
    const text = plainTextFromContent(block.content);
    let from = 0;
    while (from <= text.length) {
      const index = text.indexOf(item.sourceText, from);
      if (index < 0) break;
      matches.push({ block, start: index, end: index + item.sourceText.length });
      from = index + Math.max(item.sourceText.length, 1);
      if (matches.length > 1) return { enabled: false, reason: "ambiguous" };
    }
  }

  if (matches.length !== 1) return { enabled: false, reason: "missing" };
  const match = matches[0]!;
  return { enabled: true, blockId: match.block.id, start: match.start, end: match.end };
}

Because the stored anchor is text rather than a block id, moving the target lower in the document should not disable the edit. The new test keeps an edit enabled when matching text moves lower in the document verifies that behavior.

2. Preview no longer mutates the draft

Preview focuses the target block and scrolls it into view. It does not call insertBlocks. The visible change detail lives in the dock diff.

export function previewEditSuggestion(editor: WritingEditor, item: EditSuggestion): boolean {
  const status = getEditSuggestionStatus(editor, item);
  if (!status.enabled) return false;
  try {
    editor.setTextCursorPosition(status.blockId, "start");
  } catch {
    return false;
  }
  window.requestAnimationFrame(() => {
    const escaped = globalThis.CSS?.escape?.(status.blockId) ?? status.blockId;
    document
      .querySelector(`[data-id="${escaped}"], [data-block-id="${escaped}"]`)
      ?.scrollIntoView({ block: "center", behavior: "smooth" });
  });
  return true;
}

3. Accept re-checks before replacing

Acceptance calls the same status function again. That is the key safety property in this diff: if the document changed after the button rendered, the mutation fails closed.

export function acceptEditSuggestion(editor: WritingEditor, item: EditSuggestion): boolean {
  const status = getEditSuggestionStatus(editor, item);
  if (!status.enabled) return false;
  const block = editor.document.find((candidate) => candidate.id === status.blockId);
  if (!block || block.type === "suggestionPreview") return false;
  const text = plainTextFromContent(block.content);
  const nextText = `${text.slice(0, status.start)}${item.newText}${text.slice(status.end)}`;
  const replacement = { ...block, content: nextText } as unknown as WritingPartialBlock;
  const result = editor.replaceBlocks([block.id], [replacement]);
  const acceptedBlock = result.insertedBlocks[0];
  if (acceptedBlock) {
    try {
      editor.setTextCursorPosition(acceptedBlock.id, "end");
    } catch {
      // Focus restoration is best-effort; the edit itself has already been applied.
    }
  }
  return true;
}

One consequence is intentional deletion support: newText may be an empty string, so the splice can remove sourceText.

How disabled status reaches the UI

The inbox reducer types now allow a transient disabledReason on live entries, pinned entries, and workspace pins. The persisted suggestion state is not extended in this diff; the disabled value is derived in the renderer.

export type InboxEntry = PersistedInboxEntry & {
  stale: boolean;
  withdrawn: boolean;
  disabledReason?: "missing" | "ambiguous" | "unsupported";
};

useWorkspaceController decorates the controller output before returning it to App. That means queue rows, detail views, keyboard handling, and workspace pins all see the same current disabled decision.

const decorateInboxEntry = useCallback((entry: InboxEntry): InboxEntry => {
  const disabledReason = getSuggestionDisabledReason(editor, entry.item);
  return disabledReason ? { ...entry, disabledReason } : entry;
}, [editor]);

const decoratedInbox = useMemo(() => ({
  ...inbox,
  entries: inbox.entries.map(decorateInboxEntry),
  pinnedEntries: inbox.pinnedEntries.map(decoratePinnedEntry),
  selectedEntry: inbox.selectedEntry ? decorateInboxEntry(inbox.selectedEntry) : undefined,
  workspacePins: inbox.workspacePins.map(decorateWorkspacePin),
}), [decorateInboxEntry, decoratePinnedEntry, decorateWorkspacePin, inbox]);

Maintainer note

The type includes "unsupported", but the changed implementation of getSuggestionDisabledReason only returns missing or ambiguous for edit suggestions, and undefined for non-edits. Treat unsupported as reserved unless another adapter starts returning it.

Also, the custom suggestionPreview block and previewEvents module still exist. This diff removes them from the main preview controller path, but it does not delete their implementation.

Comprehension check

1. A user edits the draft so sourceText appears twice. What should preview and accept do?

Both should be unavailable through the decorated UI and keybinding guards because getEditSuggestionStatus returns { enabled: false, reason: "ambiguous" }. If accept is somehow called anyway, it re-runs the same check and returns false before mutating.

2. Why does this change use sourceText instead of storing a block id?

The code evidence is that matching scans the whole current document and tests assert the edit stays enabled when the matching text moves lower. A block id anchor would be more brittle after insertion, deletion, or reordering. The tradeoff is ambiguity when the same text appears more than once.

3. Which suggestion kinds can be accepted into the draft?

Only edit. The capability metadata gives edit supportsAccept: true; notes and diagrams have it set to false. The UI uses supportsSuggestionPreview for the preview/accept action group, which currently narrows to EditSuggestion.

4. Where would you start debugging if an enabled edit does not disappear after clicking Accept edit?

First check acceptEditSuggestion: did it return true and call replaceBlocks? Then check usePreviewController.accept: it should call documentChanged() and previewResolved(item.id, "accepted"). If the document changed but the suggestion remains, follow the inbox command path behind previewResolved.

5. What case is still risky even though accept re-checks safely?

The visible disabled state is derived during React rendering from the editor document. If the editor mutates without causing the controller to re-render, the button state could be stale. The accept function still protects the document, but the UI could momentarily offer an action that returns false.
