Workspaces now have real project and document boundaries

This change turns the previous single-workspace assumption into an explicit scope: every meaningful storage, renderer, event, and agent path now carries { projectId, documentId }. The likely intent is inferred from the diff and the new architecture note: prevent files, suggestions, durable events, and Pi sessions from crossing document boundaries once the app supports multiple projects and documents.

The shape of the change

The old model had a durable default document identity in many places. Some APIs accepted a document id, but storage files, suggestions, event streams, and agent state still mostly behaved as one workspace. The new model makes the selected document a session key.

The boundary changed first in the contracts

The public surface grew catalog and lifecycle operations. The important part is not the number of new methods; it is that operations which mutate or hydrate document state now have a scope in the schema.

src/shared/contracts.ts

const documentScope = Type.Object({ projectId: identifier, documentId: identifier }, strict);
const namedProject = Type.Object({ name: text(1_000) }, strict);
const namedDocument = Type.Object({ projectId: identifier, title: text(1_000) }, strict);

  "document.select": operation(documentScope, WorkspaceCatalogSchema),

  "document.save": operation(saveDocumentParams, DocumentSnapshotSchema),
  "source.import": operation(documentScope, Type.Union([SourceSnapshotSchema, Type.Undefined()])),

The catalog carries enough state for the renderer to render the Sidebar and to know what should be hydrated next.

src/shared/contracts.ts

export const WorkspaceCatalogSchema = Type.Object({
  projects: Type.Array(ProjectSnapshotSchema),
  documents: Type.Array(DocumentSummarySchema),
  selection: WorkspaceSelectionSchema,
}, strict);
A maintainer should read this as the new interface invariant: if an operation can touch document state, it must either receive a document scope or be wrapped by a handler that fills in the current selection.

Storage now has two jobs: enforce ownership and keep old data readable

The schema moves several tables from project/default scope to document scope. Version 6 adds document_id to sources, suggestion projections, suggestion command receipts, event outbox rows, and consumer cursors. It also adds the singleton workspace_settings row.

desktop/database.ts

export const DATABASE_VERSION = 6;

CREATE TABLE sources (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  document_id TEXT NOT NULL,
  ...
  FOREIGN KEY (project_id, document_id)
    REFERENCES documents(project_id, id) ON DELETE CASCADE
) STRICT;

CREATE TABLE workspace_settings (
  id INTEGER PRIMARY KEY CHECK (id = 1),
  selected_project_id TEXT REFERENCES projects(id) ON DELETE SET NULL,
  selected_document_id TEXT REFERENCES documents(id) ON DELETE SET NULL,
  updated_at INTEGER NOT NULL
) STRICT;

The repositories enforce the same idea at query time. A document lookup is no longer just "find id" in the new path; it is "find id owned by project".

Hydration is the storage operation where the new data flow is easiest to see: derive the stream id from the document, read document-specific sources and suggestions, and report the stream head for that document.

desktop/application/storage-operations.ts

hydrate(input?: Params<"hydrate">): WorkspaceSnapshot {
  input ??= this.selection();
  this.assertScope(input);
  const streamId = streamFor(input.documentId);
  const suggestion = this.deps.suggestions.get(input.projectId, input.documentId);
  return {
    streamId,
    coveredThroughSequence: this.deps.outbox.head(streamId),
    project: this.deps.projects.get(input.projectId),
    document: this.deps.documents.get(input.projectId, input.documentId),
    sources: this.deps.sources.list(input.projectId, input.documentId),
    suggestions: suggestion.state,
    suggestionRevision: suggestion.revision,
    agent: { status: "offline", cycleCount: 0 },
    activity: [],
  };
}

The migration has two halves

The database migration maps existing v5 rows onto the first document for each relevant project and verifies row counts afterward. File migration moves the legacy workspace files into the selected document directory through a journal, so a partial move can resume.

desktop/storage/workspace-migration.ts

journal = {
  version: 1,
  projectId,
  documentId,
  pending: [
    { from: join(legacyRoot, "draft.md"), to: target.draftPath },
    { from: join(legacyRoot, "sources"), to: target.sourcesDirectory },
    { from: join(legacyRoot, ".pi"), to: target.piDirectory },
  ].filter((move) => existsSync(move.from)),
};

while (journal.pending.length) {
  const move = journal.pending[0]!;
  mkdirSync(dirname(move.to), { recursive: true });
  if (existsSync(move.from) && !existsSync(move.to)) renameSync(move.from, move.to);
  else if (existsSync(move.from) && existsSync(move.to)) {
    throw new Error(`Workspace migration destination already exists: ${move.to}`);
  }
  journal.pending.shift();
  writeJournal(journalPath, journal);
}

The renderer treats document selection as a session switch

The Sidebar now renders projects and documents from the catalog and exposes create, rename, delete, and select actions. The controller does the careful part before selecting a new document: flush local document edits, flush queued suggestion commands, stop an active agent, clear preview state, then ask the desktop bridge to select the document.

src/workspace/useWorkspaceController.ts

const selectDocument = useCallback(async (projectId: string, documentId: string) => {
  if (!scope || switchPending || (scope.projectId === projectId && scope.documentId === documentId)) return;
  setSwitchPending(true);
  try {
    await document.flushForSwitch();
    await suggestionPersistence.flush();
    if (agent.runtime.status === "working" || agent.runtime.status === "waiting") await agent.stopAndWait();
    initializePreview();
    const next = await desktop.selectDocument({ projectId, documentId });
    setCatalog(next);
    setScope(next.selection);
  } catch (cause) {
    setSwitchError(cause instanceof Error ? cause.message : "The document could not be switched");
    setFailedSwitch({ projectId, documentId });
  } finally {
    setSwitchPending(false);
  }
}, [...]);

Several async hooks now capture the active scope and a generation counter before starting work. When the promise resolves, they only update state if the user is still on the same document session.

src/workspace/useSourceController.ts

function isCurrentScope(mounted, current, operation, generation, currentGeneration) {
  return mounted && generation === currentGeneration &&
    current?.projectId === operation.projectId &&
    current.documentId === operation.documentId;
}
This is the frontend race invariant: a late source import, agent start/stop response, save completion, or suggestion result must not apply to the wrong document after a switch.

Agent and durable events are tied to the selected document

The main process now spawns the Pi agent with the selected document workspace as its working directory and a document-specific session directory. On document selection, the old agent is stopped and killed, runtime is reset, and a new child process is started for the target scope.

desktop/main.ts

const spawnDocumentAgent = (scope: { projectId: string; documentId: string }) => {
  const documentWorkspace = join(
    userDataPath, "projects", scope.projectId, "documents", scope.documentId,
  );
  const sessionDirectory = join(documentWorkspace, ".pi", "sessions");
  agent = spawnChild(
    join(here, "agent.js"),
    [documentWorkspace, agentDir, sessionDirectory],
    AgentOperations,
    AgentChildMessageSchema,
    "agent-process",
    (message) => {
      if (generation === agentGeneration) void handleAgentMessage(message);
    },
  );
};

Inside the agent process, storage calls are automatically decorated with the active scope after agent.start. This keeps extension tools such as "list suggestions" on the same document without making every tool manually pass ids.

What future maintainers need to keep intact

Required invariants

  • Every persisted document mutation must include a project/document scope, or be filled from validated selection in the storage request handler.
  • A document id must be checked against its project before reading or writing document-owned rows.
  • Event stream ids must stay derivable as document:${documentId} everywhere: storage, replay, acknowledgement, and renderer coordination.
  • Switching documents must drain local queues before changing the selected scope.
  • The active project and active document cannot be deleted through the new UI/API path.

Edges to watch

  • Several repositories retain compatibility overloads that infer "first document in project" for older callers. That helps existing tests or old call sites, but it is dangerous for new multi-document behavior if used accidentally.
  • createStoragePaths still accepts default-project and default-document for migration. The database-generated path uses UUIDs, but path validation depends on callers not treating mutable names as identities.
  • The new UI uses prompt/confirm flows. Behaviour is implemented, but there is not much UX protection around destructive deletes beyond storage safeguards.
  • Test coverage added in this diff focuses on storage identity lifecycle and legacy file migration. The renderer switching path and agent respawn path are changed, but this diff does not add dedicated tests for those races.

Where this sits in recent history

The recent commits show a sequence that made this change possible: storage was split into application services, process contracts were validated, suggestions became storage-authoritative, durable event replay/acknowledgement was added, then compatibility policies were added for durable JSON. The current working tree builds on that by making the same boundaries document-aware instead of default-document-aware.

Check your mental model

1. A renderer hydrates document B while a durable event for document A arrives. What prevents document A's event from mutating document B?

After snapshot installation, DurableEventCoordinator.receive compares event.streamId with the installed this.streamId and returns early for mismatches. The main process also replaces the document agent using a generation check, which reduces old-process messages.

2. Why does document switching flush both the document autosave queue and suggestion persistence queue before calling selectDocument?

Those queues hold local mutations for the current scope. Switching first could leave a pending save or suggestion command able to complete after the selected scope changed. The controller calls document.flushForSwitch() and suggestionPersistence.flush(), then stops an active agent before selecting the next document.

3. What would break if sources stayed project-scoped while documents became selectable?

Hydrating one document in a project could show or replay sources imported for another document. The new schema adds document_id, and hydration calls sources.list(projectId, documentId), so the source list follows the selected document.

4. Where would you start debugging if suggestions from one document appeared in another after a switch?

Start at the scope propagation path: useSuggestionPersistence.seedHydratedState stores project/document ids, executeSuggestionCommand sends them, StorageOperations.executeSuggestionCommand calls assertScope, and SuggestionRepository reads/writes by project_id plus document_id. Also check for accidental use of repository compatibility overloads that infer the first document.

5. Which behaviour is directly covered by the new test file, and which changed path still looks under-tested?

Covered: UUID identity creation, scoped hydration rejection, active/last deletion guards, persisted selection, path traversal rejection, and legacy file migration. Less directly covered in this diff: renderer switch race handling, agent respawn per selected document, and Sidebar CRUD interactions.
