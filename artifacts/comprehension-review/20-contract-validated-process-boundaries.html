Contract-validated process boundaries

This change turns the Electron renderer, main process, storage utility process, and agent utility process into typed RPC peers with runtime validation at each crossing. The behaviour is still the Scribe desktop flow, but messages now carry explicit operations, protocol versions, success/failure envelopes, and bounded structured errors.

The change in one pass

Before this diff, src/shared/contracts.ts mostly listed channel names and broad TypeScript-only message shapes. A child RPC request used a free-form method: string; a response used one rpc.result shape with optional result and error fields.

The important behavioural change is not a new user-facing feature. It is a stricter boundary rule: process messages, IPC payloads, persisted JSON projections, and desktop events are rejected or converted to structured errors when they do not match the shared contract.

Follow one renderer save

A document save now gets checked three times before the database accepts it: preload checks the renderer input, main IPC checks again before forwarding, and storage operations validate before touching disk or SQLite.

// desktop/preload-bridge.ts
async function invoke<Name extends OperationName<typeof RendererOperations>>(
  ipcRenderer: PreloadIpcRenderer,
  operation: Name,
  channel: string,
  params?: OperationParams<typeof RendererOperations, Name>,
): Promise<OperationResult<typeof RendererOperations, Name>> {
  const definition = RendererOperations[operation];
  const input = parseOrContractError(definition.params, params, `preload.${operation}.params`);
  const result = await (input === undefined
    ? ipcRenderer.invoke(channel)
    : ipcRenderer.invoke(channel, input));
  return parseOrContractError(
    definition.result,
    result,
    `preload.${operation}.result`,
  ) as OperationResult<typeof RendererOperations, Name>;
}
// desktop/ipc-routing.ts
const register = <Name extends OperationName<typeof RendererOperations>>(
  operation: Name,
  channel: string,
  handler: (event: MainInvokeEvent, params: OperationParams<typeof RendererOperations, Name>) =>
    OperationResult<typeof RendererOperations, Name> | Promise<OperationResult<typeof RendererOperations, Name>>,
) => {
  ipcMain.handle(channel, async (event, ...args) => {
    if (!validateSender(event.sender)) throw new Error("Unknown renderer");
    try {
      const params = parseOrContractError(
        RendererOperations[operation].params,
        args[0],
        `main-ipc.${operation}.params`,
      ) as OperationParams<typeof RendererOperations, Name>;
      return parseOrContractError(
        RendererOperations[operation].result,
        await handler(event, params),
        `main-ipc.${operation}.result`,
      );
    } catch (error) {
      logger.error(`Renderer operation failed: ${operation}`, error);
      const contract = toContractError(error);
      const external = new Error(contract.message) as Error & { contract?: typeof contract };
      external.name = contract.code;
      external.contract = contract;
      throw external;
    }
  });
};

The storage implementation still preserves the previous consistency behaviour: file write first, transaction second, repair if the transaction fails. The new part is that StorageOperationContracts["document.save"].params is used inside the operation itself.

// desktop/storage/operations.ts
private async performDocumentSave(params: unknown): Promise<DocumentSnapshot> {
  const input = parseOrContractError(
    StorageOperationContracts["document.save"].params,
    params,
    "storage.document.save.params",
  );
  if (input.documentId !== this.deps.documentId) throw new Error("Invalid document identity");

  const current = this.deps.documents.get(this.deps.documentId);
  if (input.expectedRevision !== current.revision) throw new Error("DOCUMENT_REVISION_CONFLICT");
  if (
    JSON.stringify(input.blocks) === JSON.stringify(current.blocks) &&
    input.markdown === current.markdown
  ) return current;

  await this.deps.files.writeDraft(input.markdown);
  // transaction and repair path follow...
}

Agent/storage forwarding changed shape

The agent can still ask storage for suggestion and seed operations, but the message no longer says method and no longer receives a generic storage.result. The agent sends a storage.request with operation, and main replies with either storage.success or storage.failure.

Agent side

this.postMessage({
  kind: "storage.request",
  protocolVersion: PROTOCOL_VERSION,
  id,
  operation,
  params,
} as StorageForwardRequest);

Main process reply

response = {
  kind: "storage.failure",
  protocolVersion: PROTOCOL_VERSION,
  id: message.id,
  operation: message.operation,
  error: toContractError(error),
} as StorageForwardResult;

ChildRpc also correlates by both id and operation. A response with the right ID but the wrong operation is ignored rather than completing the wrong promise.

// desktop/child-rpc.ts
private handleRpcResult(message: Extract<ChildMessage, { kind: "rpc.success" | "rpc.failure" }>) {
  const request = this.pending.get(message.id);
  if (!request || request.operation !== message.operation) return;
  this.pending.delete(message.id);
  if (message.kind === "rpc.failure") {
    request.reject(new RemoteContractError(message.error));
    return;
  }
  try {
    request.resolve(parseOrContractError(
      request.resultSchema,
      message.result,
      `${this.boundary}.${request.operation}.result`,
    ));
  } catch (error) {
    request.reject(error instanceof Error ? error : new Error("Invalid RPC result"));
  }
}

Errors are now contracts too

The old code often sent an error string. The new boundary error has a code, message, retry flag, and bounded primitive details. Two domain errors are mapped to stable retryable errors; everything else becomes a generic internal error.

export function toContractError(error: unknown): ContractError {
  if (error instanceof ContractValidationError) return error.contract;
  if (typeof error === "object" && error !== null && "contract" in error && Check(ContractErrorSchema, error.contract)) {
    return error.contract as ContractError;
  }
  if (error instanceof Error && error.message === "DOCUMENT_REVISION_CONFLICT") {
    return { code: "DOCUMENT_REVISION_CONFLICT", message: "The document changed before it could be saved", retryable: true };
  }
  if (error instanceof Error && error.message === "STALE_SUGGESTION_REVISION") {
    return { code: "STALE_SUGGESTION_REVISION", message: "The suggestion targets an older document revision", retryable: true };
  }
  return { code: "INTERNAL_ERROR", message: "The operation could not be completed", retryable: false };
}

The validation helper deliberately avoids echoing rejected payloads. It records the boundary and up to five short issue strings.

const clean = (value: string, max = 160) => value.replace(/[\r\n\t]+/g, " ").slice(0, max);
export function parseOrContractError<Schema extends TSchema>(
  schema: Schema,
  value: unknown,
  boundary: string,
): Static<Schema> {
  if (Check(schema, value)) return value as Static<Schema>;
  const [, errors] = Errors(schema, value);
  const details: Record<string, string> = { boundary: clean(boundary, 100) };
  errors.slice(0, 5).forEach((issue, index) => {
    details[`issue${index + 1}`] = clean(`${issue.instancePath || "/"}: ${issue.message}`);
  });
  throw new ContractValidationError({
    code: "CONTRACT_VALIDATION_FAILED",
    message: `Invalid data at ${clean(boundary, 100)}`,
    retryable: false,
    details,
  });
}

Risk to remember: malformed child messages are logged or sent to stderr and then ignored. If a child sends a malformed response for a pending call, this code does not add a timeout; that request remains pending until the process exits or is disposed.

Maintenance notes for the next change

When adding an operation

Add it to the right registry in src/shared/contracts.ts. Then update routing, transport tests, and any channel mapping. The tests now expect registry inventory to match fixtures.

When changing a message shape

Update both schema unions and derived types. Check every producer includes protocolVersion: PROTOCOL_VERSION, and every consumer parses with the matching schema.

When debugging a hung RPC

Start by checking whether a response was rejected by validation or ignored because the id/operation pair did not match a pending request.

When debugging hidden data loss fears

Look at toContractError and parseOrContractError. They intentionally avoid exposing rejected payload content, so logs may contain less raw data than before.

Comprehension check

1. A renderer calls saveDocument without expectedRevision. Where is the first rejection expected?

In desktop/preload-bridge.ts. The generic invoke helper validates params against RendererOperations["document.save"].params before it calls ipcRenderer.invoke.

2. Why did the response shape split into rpc.success and rpc.failure?

The code now validates result and error paths with different schemas. A success carries a typed result; a failure carries a structured ContractError. This avoids the old ambiguous rpc.result shape where both result and error were optional.

3. What would break if a new storage operation were added only to createStorageRequestHandler?

Callers could not safely construct or validate it through the shared operation registry. The operation would be missing from StorageOperations, the RPC request/result schemas, derived operation types, and the inventory tests.

4. An agent storage request fails because of a stale document revision. What should the agent receive?

Main catches the storage call failure in createAgentMessageHandler, converts it through toContractError, and posts storage.failure. For STALE_SUGGESTION_REVISION, the error has code STALE_SUGGESTION_REVISION, a safe message, and retryable: true.

5. Where would you start debugging if an RPC promise never resolves after a child sent a response?

Check whether the response matched the expected schema and protocol version, then check whether ChildRpc.handleRpcResult found the same id and operation. Invalid child messages are ignored after logging, and mismatched results do not settle the pending request.
