The mental model to keep

The suggestion domain used to be split across types.ts, validation.ts, and a local TypeBox schema inside desktop/scribe-extension.ts. The new shape is simpler to reason about: each kind has one concrete TypeBox schema, and every runtime parser or type guard derives from that source.

Domain source

schema.ts defines schemas, TypeScript types, parser results, kind helpers, and capabilities.

Compatibility layer

types.ts and validation.ts now re-export the schema-backed API, so callers keep familiar imports.

Behavior gates

Preview, visual rendering, workspace placement, and initial card size now read from SUGGESTION_CAPABILITIES.

Inference from the diff: the main intent appears to be reducing schema drift between agent tools, desktop storage, mocks, and renderer code. No issue text was present, so that intent is inferred from the consolidation and the new tests around "family drift".

The new center of gravity: src/suggestions/schema.ts

The schema file defines the accepted payload for each suggestion family. Common fields are still shared, but family-specific fields are now expressed as concrete schemas with additionalProperties: false.

const strictObject = { additionalProperties: false } as const;
const nonEmptyString = (maxLength: number) =>
  Type.String({ minLength: 1, maxLength, pattern: "\\S" });

const commonSuggestionProperties = {
  id: nonEmptyString(200),
  dedupeKey: nonEmptyString(200),
  title: nonEmptyString(200),
  summary: nonEmptyString(1_000),
  body: nonEmptyString(8_000),
  sourceLabels: Type.Array(nonEmptyString(200), { maxItems: 12 }),
  createdAt: Type.Number(),
};

The old validator checked that common strings were non-empty and that the family-specific field existed. The new schema adds maximum lengths, rejects empty source labels, caps arrays, and rejects extra properties. That is a behavioral change at every boundary that calls parseSuggestionItem or Check.

The parser selects a concrete schema when it can

export function parseSuggestionItem(value: unknown): SuggestionParseResult {
  const kind =
    typeof value === "object" && value !== null && "kind" in value
      ? value.kind
      : undefined;
  const schema =
    typeof kind === "string" &&
    Object.prototype.hasOwnProperty.call(SUGGESTION_SCHEMAS_BY_KIND, kind)
      ? SUGGESTION_SCHEMAS_BY_KIND[kind as SuggestionKind]
      : SuggestionItemSchema;
  if (Check(schema, value)) {
    return { success: true, value: value as SuggestionItem };
  }
  const [, errors] = Errors(schema, value);
  return { success: false, issues: errors.map(safeIssue) };
}

That branch matters for error quality. A known kind is validated against its concrete schema, so a bad outline node can report /nodes/0/children/0/label instead of only saying that a union failed. Unknown kinds still fall back to the full union.

Capabilities replace repeated kind tests

export const SUGGESTION_CAPABILITIES = {

    supportsWorkspacePlacement: true, initialWorkspaceSize: { width: 320, height: 240 } },
  outline: { family: "structure", supportsPreview: false, supportsVisualRendering: true,

  mindMap: { family: "mindMap", supportsPreview: false, supportsVisualRendering: true,
    supportsWorkspacePlacement: true, initialWorkspaceSize: { width: 460, height: 340 } },

The values currently preserve existing behavior: text suggestions can preview in the document, visual suggestions render in their own presentation, and every kind can be placed on the workspace. The difference is that future kinds can be added by filling out the capability row instead of updating several unrelated branches.

Follow one suggestion through the desktop boundary

The same validation path now appears in three places that accept or reload suggestions.

Pi tool params scribe-extension.ts validates tool input schema, then adds generated fields.
Development IPC ipc-routing.ts parses before sending to storage.
Storage mutation storage.ts parses before create/update.
Hydration persisted entries, pinned entries, and workspace pins are checked before returning state.

Agent-created suggestions now round-trip through the application parser

export function toSuggestion(
  input: SuggestionToolInput | SuggestionToolUpdateInput,
  id: string = "id" in input ? input.id : randomUUID(),
  createdAt: number = Date.now(),
): SuggestionItem {
  const parsed = parseSuggestionItem({ ...input, id, createdAt });
  if (parsed.success) return parsed.value;
  throw new Error(
    `Invalid suggestion: ${formatSuggestionValidationIssues(parsed.issues)}`,
  );
}

The create tool schema omits id and createdAt; toSuggestion generates them. The update tool schema includes id and omits createdAt; toSuggestion keeps the supplied id and assigns a fresh timestamp. That timestamp behavior existed in the previous helper as createdAt: Date.now(), so the refactor keeps the observable update timestamp behavior.

Storage validates both incoming mutations and persisted projections

function assertSuggestionProjectionItems(value: unknown): asserts value is PersistedSuggestionState {
  if (typeof value !== "object" || value === null) {
    throw new Error("Invalid suggestion projection");
  }
  const state = value as Partial<PersistedSuggestionState>;
  const collections = [state.entries, state.pinnedEntries, state.workspacePins];
  if (collections.some((collection) => !Array.isArray(collection))) {
    throw new Error("Invalid suggestion projection");
  }
  for (const collection of collections as Array<Array<{ item?: unknown }>>) {
    for (const entry of collection) {
      parsedSuggestion(entry?.item, "Invalid suggestion projection item");
    }
  }
}

This is the broadest behavior change. getSuggestionState() now validates hydrated state, and saveSuggestionState() validates entries, pinned entries, and workspace pins before trimming and writing. Previously, saveSuggestionState() only checked that entries and pinnedEntries were arrays; workspacePins items were not checked there.

Development IPC reports schema issues instead of a generic failure

register(DEVELOPMENT_SUGGESTION_CHANNEL, (_event, item) => {
  const parsed = parseSuggestionItem(item);
  if (!parsed.success) {
    throw new Error(
      `Invalid development suggestion: ${formatSuggestionValidationIssues(parsed.issues)}`,
    );
  }
  return storage.call("development.suggestion.create", { item: parsed.value });
});

Invalid dev suggestions now include paths and schema keywords in the error. The same formatting is used in storage and the Pi extension.

Renderer behavior moves from kind names to capabilities

The UI changes are small in the diff, but they change the dependency direction. Components no longer ask "is this a text suggestion?" when the behavior they need is preview support. They ask the capability question directly.

Where this sits in recent history

a3def0f introduced the desktop boundary routing split. This change builds on that by validating suggestions at IPC and storage edges instead of accepting broad casts.

f2a6258 added suggestion save queue and retry behavior. The current change tightens what can be saved or rehydrated through that persistence path.

Things future maintainers should check before changing this again

Strict schemas can reject old or loose payloads. The old isSuggestionItem allowed extra properties and had fewer limits. Now extra fields, empty source labels, too many labels, overly long strings, and mixed family fields fail validation.

Persistence risk

If existing databases contain suggestion items with fields that the new schema rejects, hydrate() can fail through getSuggestionState(). The diff does not include a migration for older malformed suggestion payloads.

Agent/tool risk

Agent tool calls that include generated fields on create, or harmless-looking extra fields, may now be rejected by TypeBox before storage sees them.

Exhaustiveness invariant

Adding a new suggestion kind means adding its schema, the SUGGESTION_SCHEMAS_BY_KIND entry, its capability row, and tests. The satisfies Record<SuggestionKind, ...> checks help, but runtime behavior still depends on the capability values.

UI invariant

Preview code assumes that supportsSuggestionPreview(item) narrows to a text suggestion with insertText. If a future non-text kind supports preview, the preview insertion path must change first.

Comprehension check

Use these to confirm the code path is in your head before reviewing or modifying it.

1. An agent calls create_suggestion with a valid snippet payload but includes an extra debug property. What happens?

The tool input schema should reject it because schemas are strict with additionalProperties: false. If it reached toSuggestion, parseSuggestionItem would also reject the extra property.

2. Why does parseSuggestionItem choose SUGGESTION_SCHEMAS_BY_KIND[kind] before validating?

For known kinds, validating against the concrete schema produces more useful field paths and avoids generic union errors. This is tested by the nested outline-node path check.

3. A maintainer wants to add a suggestion kind that can be placed on the workspace but cannot preview in the editor. Which part of the new design should they update?

They need a concrete schema, a mapping entry in SUGGESTION_SCHEMAS_BY_KIND, and a capability row with supportsWorkspacePlacement: true and supportsPreview: false. Tests should add a valid payload for the new kind and capability expectations.

4. What changed about persisted suggestion state validation?

getSuggestionState() now validates every item in entries, pinnedEntries, and workspacePins. saveSuggestionState() also validates all three collections. Previously, the save path only checked that entries and pinnedEntries were arrays.

5. Where would you start debugging if the "Preview in document" button disappeared for snippets?

Start with SUGGESTION_CAPABILITIES.snippet.supportsPreview, then the supportsSuggestionPreview helper. The button, keyboard action, and handlePreview now depend on that capability instead of directly checking isTextSuggestion.
