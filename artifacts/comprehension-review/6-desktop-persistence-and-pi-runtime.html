Desktop persistence turns the writing partner into a second runtime

This change adds an Electron path beside the existing browser development path. The browser still uses the injected mock suggestion feed, while the desktop app hydrates from SQLite, autosaves accepted document content, imports source files, and runs a Pi-powered background agent in a utility process.

Review target: current working tree Verification run: npm test -- --run, npm run build Relevant history: mock feed, pins, docs, then desktop runtime

The new shape of the app

The most important mental model is that App.tsx now chooses between two producers and two persistence stories at startup. If window.scribe exists, the renderer is running under Electron and talks through the preload bridge. If it does not, the Vite browser flow stays in memory and keeps using the mock suggestion controller.

Preload bridgetyped window.scribe API
Electron mainIPC, windows, scheduler
Storage processSQLite, source text, outbox

The arrows are bidirectional in the implementation: renderer commands go through preload/main, storage emits committed events back through main, and the agent asks storage to read content and commit suggestions.

The bridge is the new boundary

The preload does not expose Electron primitives. It exposes the application operations in DesktopBridge, which becomes the contract between renderer and desktop services.

export type DesktopBridge = {
  hydrate(): Promise<WorkspaceSnapshot>;
  saveDocument(input: {
    documentId: string;
    blocks: unknown[];
    expectedRevision: number;
  }): Promise<DocumentSnapshot>;
  saveSuggestionState(state: PersistedSuggestionState): Promise<void>;
  importSource(): Promise<SourceSnapshot | undefined>;
  setProvider(input: ProviderSettings & { apiKey: string }): Promise<ProviderSettings>;
  setAgentPaused(paused: boolean): Promise<AgentRuntime>;
  considerNow(): Promise<void>;
  subscribe(listener: (event: DesktopEvent) => void): () => void;
};

The implementation in desktop/preload.ts maps each method to a named IPC channel and wires subscribe to scribe:event. The global type in src/vite-env.d.ts lets the React side safely check window.scribe.

export function getDesktopBridge(): DesktopBridge | undefined {
  return window.scribe;
}

export function createDesktopSuggestionFeed(
  bridge: DesktopBridge,
): SuggestionFeed {
  return {
    subscribe(listener) {
      return bridge.subscribe((event: DesktopEvent) => {
        let suggestionEvent: SuggestionEvent | undefined;
        if (event.type === "suggestion.event") {
          suggestionEvent = event.event;
        } else if (event.type === "agent.runtime") {
          suggestionEvent = event.runtime.lastError
            ? { type: "agent.error", message: event.runtime.lastError, recoverable: true }
            : { type: "agent.status", status: event.runtime.running
                ? "working"
                : event.runtime.configured ? "idle" : "offline" };
        }
        if (suggestionEvent) listener(suggestionEvent);
      });
    },
  };
}

That adapter is deliberately narrow: the inbox still receives normal SuggestionEvent values. This preserves the reducer path introduced by the earlier writing-partner refactor and avoids putting Electron logic into SuggestionDock.

Follow desktop startup once

On the renderer side, App.tsx creates either the desktop feed or the injected browser feed. In desktop mode it also hydrates the saved workspace and passes an onStateChange callback into the inbox hook so suggestion projection changes can be persisted.

const desktop = useMemo(() => getDesktopBridge(), []);
const feed = useMemo(
  () =>
    desktop
      ? createDesktopSuggestionFeed(desktop)
      : createInjectedSuggestionFeed(),
  [desktop],
);

const inboxOptions = useMemo(
  () => ({ onStateChange: desktop ? saveSuggestionState : undefined }),
  [desktop, saveSuggestionState],
);
const inbox = useSuggestionInbox(feed, inboxOptions);

The hydration effect is where seeded browser content gives way to the stored document. It replaces the BlockNote document only if storage returns saved blocks, restores provider/runtime/source state, and hydrates the reducer-owned suggestion projection.

void desktop
  .hydrate()
  .then((snapshot) => {
    hydrationInProgressRef.current = true;
    if (snapshot.document.blocks.length) {
      editor.replaceBlocks(
        editor.document,
        snapshot.document.blocks as WritingPartialBlock[],
      );
    }
    documentIdRef.current = snapshot.document.id;
    documentRevisionRef.current = snapshot.document.revision;
    setProvider(snapshot.provider);
    setAgentRuntime(snapshot.agent);
    setSources(snapshot.sources);
    hydrateInbox(snapshot.suggestions);
    window.requestAnimationFrame(() => {
      hydrationInProgressRef.current = false;
      documentHydratedRef.current = true;
    });
  })

Why the hydration flags matter: editor.replaceBlocks can trigger editor-change plumbing. The flags keep startup replacement from being immediately written back as a user edit. The save path also checks documentHydratedRef.

Autosave and suggestion projection are separate

Document persistence writes accepted BlockNote content after a 650 ms debounce. Before saving, the renderer filters out suggestionPreview blocks, so a temporary preview is not treated as accepted draft content after restart.

const persistDocument = useCallback(() => {
  if (!desktop || !documentHydratedRef.current || hydrationInProgressRef.current) {
    return;
  }
  const blocks = editor.document.filter(
    (block) => block.type !== "suggestionPreview",
  );
  void desktop
    .saveDocument({
      documentId: documentIdRef.current,
      blocks,
      expectedRevision: documentRevisionRef.current,
    })
    .then((document) => {
      documentRevisionRef.current = document.revision;
    });
}, [desktop, editor]);

Suggestion state persists through the reducer hook instead of through the editor. The reducer now has a hydrate action and a serializer that leaves out ephemeral fields such as activePreviewId, status, and error.

if (action.type === "hydrate") {

}

export function persistedSuggestionState(

}

The new reducer test locks down the important part of this behaviour: durable entries and z-index restore, but preview state does not.

Where SQLite owns truth

desktop/storage.ts is the only SQLite owner. It creates a default project/document, stores source text, stores the visible suggestion projection, records agent runs/transcripts, and commits outbound events. Mutating operations use transaction, and the outbox is flushed only after commit.

function transaction<T>(work: () => T): T {
  db.exec("BEGIN IMMEDIATE");
  try {
    const result = work();
    db.exec("COMMIT");
    flushOutbox();
    return result;
  } catch (error) {
    db.exec("ROLLBACK");
    throw error;
  }
}

Agent-created suggestions show that ordering. Storage checks the document revision supplied by the agent, applies dedupe, updates the durable projection, and emits the existing reducer event type.

function createSuggestion(params: unknown) {
  const input = params as { item: SuggestionItem; expectedDocumentRevision: number };
  const document = getDocument();
  if (input.expectedDocumentRevision !== document.revision) {
    throw new Error("The target document changed; start a new observation");
  }
  return transaction(() => {
    const state = getSuggestionState();
    if (state.seenKeys[input.item.dedupeKey]) return { accepted: false };
    state.seenKeys[input.item.dedupeKey] = true;
    state.entries.push({ item: input.item, viewed: false, stale: false, withdrawn: false });
    enforceSuggestionLimit(state);
    putSuggestionState(state);
    emitSuggestion({ type: "suggestion.added", item: input.item });
    return { accepted: true };
  });
}

Source import follows the same storage boundary. The renderer only gets SourceSnapshot values for the sidebar; extracted file content remains in storage for agent tools.

The Pi agent path

desktop/main.ts owns the observation schedule. It asks storage for an ObservationSeed, refuses to run when the provider is disabled or paused, and skips repeated observations of the same project revision unless the user forces a run.

async function observe(force = false) {
  const seed = await storage.call<ObservationSeed>("agent.seed");
  if (!seed.provider.enabled || seed.paused) {
    setRuntime({
      configured: seed.provider.enabled,
      paused: seed.paused,
      running: false,
    });
    return;
  }
  if (!force && seed.projectRevision === lastCompletedProjectRevision) return;
  agent.post({ kind: "observe", seed, apiKey, force });
}

The agent process builds Pi tools that operate through storage RPC. A created suggestion does not edit the document directly; it asks storage to commit a suggestion item for the current document revision.

const create: AgentTool<typeof suggestionSchema> = {
  name: "create_suggestion",
  label: "Create suggestion",
  description: "Create a useful suggestion for the active document. This does not edit the document.",
  parameters: suggestionSchema,
  execute: async (_id, params) => {
    const item = toSuggestion(params as SuggestionInput);
    const result = await storageCall("agent.suggestion.create", {
      item,
      expectedDocumentRevision: seed.documentRevision,
    });
    return { content: [{ type: "text", text: JSON.stringify(result) }], details: { item } };
  },
};

Provider controls are exposed only in desktop mode through AgentControls inserted into SuggestionDock. The API key is kept in process memory for the current launch, while provider/model/base URL and enabled state are persisted.

What future maintainers should be careful about

Invariant or riskEvidence in the changeWhy it matters
The browser path must keep working without Electron. getDesktopBridge() returns window.scribe, and App.tsx falls back to createInjectedSuggestionFeed(). Any renderer code that assumes the bridge exists would break Vite development and tests.
Do not persist temporary previews as accepted document content. persistDocument filters block.type !== "suggestionPreview". Preview blocks are a UI draft state, not the accepted saved document.
Storage is the consistency point for agent suggestions. createSuggestion, updateSuggestion, and retractSuggestion check document revision before mutating suggestion state. An observation based on an old document should not commit suggestions against newer content.
Document autosave accepts an expectedRevision but does not currently enforce it in storage. saveDocument reads the current document and updates when blocks differ, but the shown code does not compare input.expectedRevision with current.revision. This is fine for the current single-active-document model, but it is a gap if multi-window or conflict handling is added.
Secrets are intentionally not durable. desktop/main.ts keeps apiKey in a module variable and provider.set stores only provider/model/base URL/enabled. Restarting the desktop app requires re-entering the key before enabled agent calls can authenticate.

How this relates to recent history

The last few commits created the seams this change now uses. bb071f9 introduced the writing-partner reducer/feed flow. 0b62339 added pins and workspace card geometry. 4c96792 and 9538f0b consolidated browser mock event injection and documented the client-only state. This desktop change keeps the reducer and feed interface, then swaps in an Electron-backed producer and storage layer.

That history explains why the change mostly avoids rewriting suggestion UI. The new work is concentrated at the boundaries: DesktopBridge, createDesktopSuggestionFeed, storage/agent utility processes, and the renderer hydration/autosave glue.

Comprehension checks

  1. When the app is opened in Vite without Electron, which suggestion feed is used, and why?

    Answer

    createInjectedSuggestionFeed() is used because getDesktopBridge() returns window.scribe, which is undefined outside the Electron preload path. App.tsx branches on that value.

  2. Why does document autosave filter out suggestionPreview blocks?

    Answer

    Those blocks represent an editable preview of a suggestion before the user accepts it. Saving them with normal document blocks would restore temporary preview UI as accepted draft content after restart.

  3. An agent run creates a suggestion, but the user edited the document after the run started. Which layer rejects the stale write?

    Answer

    desktop/storage.ts rejects it in createSuggestion, updateSuggestion, or retractSuggestion by comparing expectedDocumentRevision with the current document revision.

  4. What would break if useSuggestionInbox persisted state before hydration finished?

    Answer

    The initial empty reducer state could overwrite the stored suggestion projection. The hook starts with hydrated false when onStateChange exists, then sets it true after hydrate dispatches the durable state.

  5. Where would you start debugging if the desktop sidebar shows an imported source but the agent cannot search it?

    Answer

    Start in desktop/storage.ts: importSource should copy the file, extract text, persist sources.extracted_text, and update project revision. Then check agent.content.search, which calls listProjectContent() and readProjectContent() against stored text.
