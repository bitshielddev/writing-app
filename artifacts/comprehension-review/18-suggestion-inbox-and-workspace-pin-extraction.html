This worktree mostly turns two large modules into smaller units. There is one user-visible-ish persistence change to watch: the inbox hook now tries to avoid writing persisted state when only ephemeral UI state changes.

The shape of the change

Two previous entry files are kept as compatibility surfaces, while their implementation moved behind more focused modules. src/suggestions/inbox.ts is now a barrel for reducer and hook exports. src/components/WorkspacePins.tsx is now a barrel for the extracted workspace pin component tree.

src/suggestions/inboxReducer.ts

Owns reducer actions, durable-state projection, and selector helpers for sorted entries, selected entry, and unread count.

src/suggestions/useSuggestionInbox.ts

Owns React subscription/effect code and dispatch callbacks. It depends on reducer selectors instead of embedding all state logic.

src/components/workspacePins/*

Splits geometry policy, bounds repair, pointer/keyboard interaction, card rendering, and list rendering.

DocumentEditor.tsx

Uses the shared initial-placement geometry helper instead of duplicating workspace-card size and cascade math.

scripts/import-cycles.test.mjs

Adds a small dependency-cycle guard for the newly extracted modules and earlier workspace service hooks.

The inbox split: reducer remains pure, hook owns React effects

The old inbox.ts combined action reduction, sorting, persistence projection, feed subscription, hydration, and callback creation. The new boundary separates the state machine from React.

The reducer file still contains the familiar behavior: dedupe suggestions, protect selected or previewed entries from queue trimming, freeze pinned snapshots, move pins into the workspace, and clear active previews on resolution. The extra exports make those derived views explicit. 

export function projectPersistedSuggestionState(
  state: InboxState,
): PersistedSuggestionState {
  return {
    entries: state.entries,
    pinnedEntries: state.pinnedEntries,
    workspacePins: state.workspacePins,
    seenKeys: state.seenKeys,
    nextZIndex: state.nextZIndex,
  };
}

export const persistedSuggestionState = projectPersistedSuggestionState;

export function selectSortedInboxEntries(state: InboxState): InboxEntry[] {
  return [...state.entries].sort((left, right) => {
    if (left.viewed !== right.viewed) return left.viewed ? 1 : -1;
    return right.item.createdAt - left.item.createdAt;
  });
}

Why this matters for maintenance

The hook can now be tested for subscription and persistence behavior without retesting every reducer transition. The reducer can be imported directly by tests and other non-React code without pulling in React hooks.

The old import path still works because inbox.ts re-exports the public names, including useSuggestionInbox and the reducer types.

The persistence behavior to keep in your head

useSuggestionInbox now builds a memoized durable state object from only the fields that should be persisted. That excludes selectedId and activePreviewId. The effect writes only after hydration, as before, but it now depends on durableState rather than the entire reducer state. 

const { entries, nextZIndex, pinnedEntries, seenKeys, workspacePins } = state;
const durableState = useMemo(
  () => ({ entries, pinnedEntries, workspacePins, seenKeys, nextZIndex }),
  [entries, nextZIndex, pinnedEntries, seenKeys, workspacePins],
);

useEffect(() => {
  if (hydrated) onStateChange?.(durableState);
}, [durableState, hydrated, onStateChange]);

What still persists

Selecting an unread suggestion can still write, because select marks entries as viewed. Viewed state is durable, so the entries array changes and durableState changes.

What should not persist

A plain back transition should not write when it only clears selectedId. The reducer now avoids replacing entries if no withdrawn detail entry was removed. 

case "back": {
  const entries = state.entries.filter(
    (entry) => !entry.withdrawn || entry.item.id === state.activePreviewId,
  );
  return {
    ...state,
    selectedId: undefined,
    entries: entries.length === state.entries.length ? state.entries : entries,
  };
}

The new useSuggestionInbox.test.ts locks this down by expecting no extra onStateChange call after back() in a case where nothing durable changed.

The geometry policy became shared code

DocumentEditor no longer performs its own width, height, and cascade calculations inline. It asks createInitialWorkspacePinRect to produce the first rectangle from a preferred size, canvas bounds, visible top offset, and stack index. 

return createInitialWorkspacePinRect({
  preferredSize: baseSize,
  bounds: { width: canvas.clientWidth, height: canvas.clientHeight },
  visibleTop,
  stackIndex,
});

That helper uses the same minimum size and edge-padding policy as drag, resize, and bounds repair. 

export const WORKSPACE_PIN_GEOMETRY_POLICY = {
  edgePadding: 16,
  minimumWidth: 280,
  minimumHeight: 180,
  cascadeStep: 24,
  cascadeCount: 5,
} as const;

export function clampWorkspacePinRect(
  rect: WorkspacePinRect,
  bounds: WorkspacePinBounds,
): WorkspacePinRect {
  const maxWidth = Math.max(minimumWidth, bounds.width - edgePadding * 2);
  const maxHeight = Math.max(minimumHeight, bounds.height - edgePadding * 2);
  const width = clamp(rect.width, minimumWidth, maxWidth);
  const height = clamp(rect.height, minimumHeight, maxHeight);
  return {
    x: clamp(rect.x, edgePadding, bounds.width - width - edgePadding),
    y: clamp(rect.y, edgePadding, bounds.height - height - edgePadding),
    width,
    height,
  };
}

Small caveat in the policy

createInitialWorkspacePinRect still uses literal 24 offsets for the initial x/y placement. The cascade step is named in policy, but the initial inset is not. If a future change wants all spacing controlled by policy, that is one remaining hard-coded value to look for.

Where the behavior can still surprise you

Repair timing moved to an animation frame

Bounds repair is now scheduled through window.requestAnimationFrame. That is consistent with layout measurement work, but tests or callers that expected synchronous repair writes after a resize notification will need to wait a frame.

The cycle test is a guardrail, not a full import parser

scripts/import-cycles.test.mjs scans a fixed list of target modules with a regex. It is useful for these extraction boundaries, but it will not find every possible project cycle unless the list is kept current. 

const frame = window.requestAnimationFrame(() => {
  const repairs = pinsRef.current.flatMap((pin) => {
    if (pin.pendingInitialPlacement) return [];

Comprehension check

1. Why does useSuggestionInbox build durableState instead of passing the full reducer state to onStateChange?

Because selectedId and activePreviewId are ephemeral UI state. The hook should persist entries, pinned entries, workspace pins, seen keys, and z-index state, but not every selection or preview pointer change.

2. A user selects an unread suggestion and then presses back. Which part should persist, and which part should not?

Selecting can persist because it marks the entry as viewed, which is durable. Pressing back should not persist if it only clears selectedId and does not remove a withdrawn entry.

3. Where would you start debugging if a workspace card can be dragged outside the canvas after this refactor?

Start in src/components/workspacePins/geometry.ts, specifically clampWorkspacePinRect, then check that useWorkspacePinInteraction calls updateDraft for drag and resize and receives correct bounds.

4. What would break if pending initial pins were included in the bounds repair loop?

A pin awaiting its first measured placement could be repaired using stale or placeholder geometry before DocumentEditor writes the intended initial rectangle. The current code skips pendingInitialPlacement in rendering and repair paths.

5. What does the new import-cycle test protect, and what is its limitation?

It protects the named extraction modules from depending on each other in a cycle. Its limitation is scope: it only checks targetModules and uses a simple import regex, so new modules must be added deliberately.
