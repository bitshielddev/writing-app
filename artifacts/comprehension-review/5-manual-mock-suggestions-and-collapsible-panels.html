The channel deliberately narrows what can cross the boundary

mockSuggestionChannel.ts accepts only suggestion.added events, and only when the payload passes isSuggestionItem. Other event types such as agent.status, suggestion.updated, and suggestion.retracted are no longer produced by the mock feed. 

function isAddedSuggestionEvent(value: unknown): value is AddedSuggestionEvent {
  if (typeof value !== "object" || value === null) {
    return false;
  }

  const candidate = value as { type?: unknown; item?: unknown };
  return (
    candidate.type === "suggestion.added" && isSuggestionItem(candidate.item)
  );
}

The shape check is important because BroadcastChannel receives untyped data. The tests exercise this directly by posting { type: "suggestion.added", item: { title: "Bad" } } through a fake channel and expecting no inbox events.

Draft building encodes the supported suggestion families

The controller form is broad enough to create all current suggestion kinds, but the required fields differ by kind. Text suggestions require insertText. Structure suggestions require recursive nodes. Mind maps require Mermaid source plus an accessible description. 

if (draft.kind === "snippet" || draft.kind === "fact" || draft.kind === "term") {
  return {
    ...common,
    kind: draft.kind,
    insertText: required(draft.insertText, "Insert text"),
  };

if (draft.kind === "outline" || draft.kind === "layout") {
  let nodes: unknown;
  try {
    nodes = JSON.parse(required(draft.nodes, "Nodes JSON"));
  } catch (error) {
    if (error instanceof SyntaxError) {
      throw new Error("Nodes must be valid JSON.", { cause: error });
    }
    throw error;
  }

  if (!isStructureNodes(nodes)) {
    throw new Error("Nodes must be a non-empty array ...");
  }

  return { ...common, kind: draft.kind, nodes };
}
Maintenance note: if a future change adds a new SuggestionKind, update the form options, buildMockSuggestion, and isSuggestionItem together. Otherwise the controller might let the user select a kind that the channel later rejects, or the app might accept a shape the form cannot produce.

What the mock feed stopped doing

The old createMockSuggestionFeed was a small scripted simulator. It emitted status changes, generated initial suggestions, reacted to steering, watched document updates, and retracted or updated suggestions on timers. The new feed ignores the context argument and only subscribes to manual injection.

Invariants future changes should preserve

  • Channel messages remain untrusted. Anything arriving from BroadcastChannel must pass a runtime shape check before becoming a SuggestionEvent.
  • Only one subscription opens the feed channel. createMockSuggestionFeed starts on the first listener and stops when the last listener unsubscribes.
  • Drawer state and panel state stay separate. Mobile buttons control dialogs; desktop buttons control grid panels.
  • Closed panels count as zero width for resize constraints. Otherwise a hidden column would still reduce the editor's available resize range.
  • Manual mock mode does not imply agent behavior. Steering and retry are no-ops here, so any test that expects generated suggestions should use a different feed or reintroduce that behavior deliberately.

Comprehension check

Use these to verify that the main paths are clear before modifying this area.

1. A developer opens the app at / and sees an empty writing partner. Is that a regression according to this diff?

No, not by itself. The mock feed no longer emits initial timed suggestions. It only emits suggestions received from the controller over BroadcastChannel.

2. What would break if subscribeToMockSuggestions forwarded every channel message without isSuggestionItem?

Untyped or malformed data from another tab could enter the inbox as a suggestion event. Components downstream expect fields such as id, kind, sourceLabels, and kind-specific fields, so rendering or preview behavior could fail at runtime.

3. Why does the resize code treat a closed opposite panel as width 0?

The resize maximum protects the editor's minimum width by subtracting neighboring panel widths. If a closed panel still counted as its minimum width, the visible panel would be unnecessarily constrained even though that neighbor is not occupying a grid track.

4. Where would you start debugging if the controller says a suggestion was sent but the dock never updates?

Start at the channel boundary: confirm the writing app tab is open on the same origin, BroadcastChannel exists, the app has a feed subscriber, and the published suggestion passes isSuggestionItem. The new tests show that malformed messages are intentionally ignored.

5. If a new suggestion kind is added, which parts of this change likely need updates together?

Update the controller's kind options and kind-specific fields, buildMockSuggestion, isSuggestionItem, and tests for valid and invalid payloads. The channel only forwards payloads accepted by the validator.
