Comprehension review

Suggestion state now has a durable history, not just a current row

This change moves suggestions toward an event-sourced storage path. Commands are wrapped with actor/version metadata, converted into immutable suggestion facts, applied through a strict reducer, and then delivered through the existing durable event stream as legacy-compatible suggestion.event payloads.

The shortest useful mental model

Before this change, storage operations mostly transformed the current suggestion projection and wrote the updated projection directly. The new path treats the projection as derived state. The durable source is a command receipt plus a sequence of suggestion facts in suggestion_event_history.

IntentWriter, agent, or development code asks for a suggestion change.
EnvelopeStorage adds actor, version, expected revisions, command id, and timestamp.
FactA pure aggregate decides whether the command emits one or more facts.
ProjectionThe strict reducer advances state only across contiguous fact sequences.
DeliveryThe outbox references the fact and materializes the older payload shape on read.

Evidence for intent: the docs say storage now records versioned intent, appends immutable facts, advances the projection, and stores the receipt in one transaction. The code and tests match that. There is no issue or PR text in the local context, so broader product motivation is inferred from these artifacts.

Follow one command through the new path

The writer command path in StorageOperations.commandSuggestion now builds a SuggestionCommandEnvelope before reading/validating the projection. The envelope is persisted with the receipt, so later diagnostics can tell which actor and command version produced the result.

const command: SuggestionCommandEnvelope = {
  commandId: input.commandId, projectId: input.projectId, documentId: input.documentId,
  actor: { type: "writer" }, version: SUGGESTION_COMMAND_VERSION,
  command: input.command as DurableSuggestionCommand,
  expectedSuggestionRevision: input.expectedSuggestionRevision,
  requestedAt: this.deps.clock.now(),
};
const current = this.deps.suggestions.get(input.projectId, input.documentId);
if (input.expectedSuggestionRevision !== current.revision) {
  const conflict = { commandId: input.commandId, status: "conflict" as const,
    suggestionRevision: current.revision, state: current.state,
    reason: "Suggestion state changed before the command was applied" };
  this.deps.suggestions.recordCommandReceipt(command, conflict, undefined,
    current.coveredThroughSequence, "SUGGESTION_REVISION_CONFLICT");
  return conflict;
}

Two details matter here:

  • The repeated-command short-circuit still happens first via findReceipt, preserving idempotent command responses.
  • Conflicts and rejections now record richer receipts through recordCommandReceipt, including sequence coverage and an optional error code.

When the command is valid, the old direct transition call has been replaced with a domain decision and an append to immutable facts.

const decision = decideSuggestionCommand(current.state, command.command);
if (decision.status === "rejected") {
  const rejected = { commandId: input.commandId, status: "rejected" as const,
    suggestionRevision: current.revision, state: current.state, reason: decision.reason };
  this.deps.suggestions.recordCommandReceipt(command, rejected, undefined,
    current.coveredThroughSequence, "SUGGESTION_COMMAND_REJECTED");
  return rejected;
}
const persisted = decision.status === "changed"
  ? this.deps.suggestions.appendFacts(command, decision.facts,
    decision.facts.map(() => this.deps.identities.next()))
  : { projection: current, events: [] };
const projection = persisted.projection;

The application layer no longer needs to know how a pin, dismiss, preview resolution, publish, update, or retract mutates the projection. It asks the domain aggregate for facts and asks the repository to persist/apply those facts.

The aggregate has two jobs: decide, then replay

The new domain module deliberately separates command validation from projection replay. decideSuggestionCommand still uses the existing suggestion transition functions, but its output is facts rather than a new projection.

export function decideSuggestionCommand(
  current: PersistedSuggestionState,
  command: SuggestionIntent,
): AggregateDecision {
  if (command.type === "publish" || command.type === "update" || command.type === "retract") {
    const event = command.type === "publish"
      ? { type: "suggestion.added" as const, item: command.item }
      : command.type === "update"
        ? { type: "suggestion.updated" as const, item: command.item }
        : { type: "suggestion.retracted" as const, id: command.suggestionId };
    const transition = applySuggestionAgentEvent(current, event);
    if (transition.status !== "changed") return transition.status === "unchanged"
      ? { status: "unchanged", facts: [] }
      : { status: "rejected", facts: [], reason: transition.reason };
    return { status: "changed", facts: [command.type === "publish"
      ? { type: "suggestion.published", version: 1, item: command.item }
      : command.type === "update"
        ? { type: "suggestion.updated", version: 1, item: command.item }
        : { type: "suggestion.retracted", version: 1, suggestionId: command.suggestionId }] };
  }

  const transition = applySuggestionCommand(current, command);
  if (transition.status !== "changed") return transition.status === "unchanged"
    ? { status: "unchanged", facts: [] }
    : { status: "rejected", facts: [], reason: transition.reason };
  return { status: "changed", facts: [factForWriterCommand(command)] };
}

The replay side is stricter than a normal reducer. It ignores already-covered events, rejects gaps, rejects unknown fact versions, and rejects redundant facts. That strictness is what makes verification and repair meaningful.

export function applySuggestionFact(
  projection: SuggestionProjectionView,
  event: Pick<SequencedSuggestionFact, "sequence" | "fact">,
): SuggestionProjectionView {
  if (event.sequence <= projection.coveredThroughSequence) return projection;
  if (event.sequence !== projection.coveredThroughSequence + 1) {
    throw new Error(`SUGGESTION_EVENT_SEQUENCE_GAP:${projection.coveredThroughSequence + 1}:${event.sequence}`);
  }
  if (event.fact.version !== SUGGESTION_EVENT_VERSION) throw new Error("UNKNOWN_SUGGESTION_EVENT_VERSION");
  if (event.fact.type === "suggestion.projectionImported") {
    if (projection.coveredThroughSequence !== 0) throw new Error("INVALID_PROJECTION_IMPORT_POSITION");
    return { state: structuredClone(event.fact.state), revision: projection.revision,
      coveredThroughSequence: event.sequence };
  }
  const transition = transitionForFact(projection.state, event.fact);
  if (transition.status !== "changed") throw new Error(
    transition.status === "rejected" ? `INVALID_SUGGESTION_EVENT:${transition.reason}` : "REDUNDANT_SUGGESTION_EVENT",
  );
  return { state: transition.state, revision: projection.revision + 1,
    coveredThroughSequence: event.sequence };
}

What actually lands in SQLite

The schema split is the center of the change. suggestion_projection keeps the current state for fast hydration, while suggestion_event_history is the append-only sequence that can rebuild that state. Receipts also now store actor, command type/version, payload, outcome, and sequence metadata.

CREATE TABLE suggestion_projection (
  project_id TEXT NOT NULL,
  document_id TEXT PRIMARY KEY,
  state_json TEXT NOT NULL,
  revision INTEGER NOT NULL DEFAULT 0,
  covered_through_sequence INTEGER NOT NULL DEFAULT 0,
  projection_version INTEGER NOT NULL DEFAULT 1,
  checksum TEXT NOT NULL DEFAULT '',
  updated_at INTEGER NOT NULL,
  FOREIGN KEY (project_id, document_id) REFERENCES documents(project_id, id) ON DELETE CASCADE
) STRICT;

CREATE TABLE suggestion_event_history (
  event_id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  document_id TEXT NOT NULL,
  sequence INTEGER NOT NULL,
  command_id TEXT NOT NULL,
  actor_json TEXT NOT NULL,
  event_type TEXT NOT NULL,
  event_version INTEGER NOT NULL,
  payload_json TEXT NOT NULL,
  occurred_at INTEGER NOT NULL,
  UNIQUE (document_id, sequence),
  FOREIGN KEY (project_id, document_id) REFERENCES documents(project_id, id) ON DELETE CASCADE
) STRICT;

The repository append path applies each new fact to the in-memory projection before it writes the final projection row. The projection update is guarded by both the previous revision and previous coverage.

Because StorageOperations calls this inside transactions.run alongside receipt recording, a failed write rolls back the history and projection together. The integration test rolls back history and projection together on transaction failure covers that behavior directly for appendFacts.

Existing consumers still see suggestion.event

The outbox schema now permits either a full event_json payload or a reference to suggestion_event_history. Suggestion facts use the reference path.

CREATE TABLE event_outbox (
  event_id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  document_id TEXT NOT NULL,
  stream_id TEXT NOT NULL,
  sequence INTEGER NOT NULL,
  event_json TEXT,
  suggestion_event_id TEXT REFERENCES suggestion_event_history(event_id) ON DELETE CASCADE,
  occurred_at INTEGER NOT NULL,
  causation_id TEXT,
  created_at INTEGER NOT NULL,
  dispatched_at INTEGER,
  CHECK (event_json IS NOT NULL OR suggestion_event_id IS NOT NULL),
  UNIQUE (stream_id, sequence),
  FOREIGN KEY (project_id, document_id) REFERENCES documents(project_id, id) ON DELETE CASCADE
) STRICT;

On enqueue, the outbox looks up the fact and stores only a reference. On read/replay, it rebuilds the projection up to that fact and returns a legacy-compatible payload.

private parseRow(row: PersistedEventRow) {
  if (!row.suggestion_event_id) {
    if (row.event_json === null) throw new Error("INVALID_DURABLE_DELIVERY_REFERENCE");
    return parseEnvelope(this.db, row);
  }
  const repository = new SuggestionRepository(this.db);
  let projection = { state: createEmptySuggestionState(), revision: 0, coveredThroughSequence: 0 };
  let delivered: SequencedSuggestionFact | undefined;
  for (const event of repository.history(row.project_id, row.document_id)) {
    projection = applySuggestionFact(projection, event);
    if (event.eventId === row.suggestion_event_id) {
      delivered = event;
      break;
    }
  }
  if (!delivered) throw new Error("SUGGESTION_DELIVERY_HISTORY_MISSING");
  const payload = parseOrContractError(DurableEventPayloadSchema, {
    type: "suggestion.event",
    event: legacySuggestionEvent(delivered.fact),
    commandId: delivered.commandId,
    suggestionRevision: projection.revision,
    state: projection.state,
  }, "persisted.suggestion-delivery.payload");
  return parseOrContractError(DurableEventEnvelopeSchema, {
    eventId: row.event_id, streamId: row.stream_id, sequence: row.sequence,
    occurredAt: row.occurred_at, causationId: row.causation_id ?? undefined, payload,
  }, "persisted.suggestion-delivery");
}

This preserves the renderer-facing contract while changing the durable source behind it. The integration test proves replay sees two suggestion events after an agent-created suggestion and a writer pin command.

Verification, checkpoints, and repair are now part of storage

The new repository can rebuild a projection from the latest checkpoint plus remaining history, compare checksums, create bounded checkpoints, and replace the projection after a backup-confirmed repair.

Verify

Rebuilds, compares checksum and revision, and quarantines failure metadata on rebuild errors.

Checkpoint

Created when forced, when coverage advances by at least 500 facts, or when replay took more than two seconds. Only ten latest checkpoints per document are retained.

Repair

Requires backupConfirmed. Service-level repair creates a database backup first and rejects :memory:.

suggestionMaintenance: {
  verify(projectId: string, documentId: string) {
    return suggestions.verify(projectId, documentId);
  },
  diagnostics(projectId: string, documentId: string) {
    return suggestions.diagnostics(projectId, documentId);
  },
  checkpoint(projectId: string, documentId: string) {
    return suggestions.createCheckpoint(projectId, documentId, 0, true);
  },
  repair(projectId: string, documentId: string) {
    if (options.databasePath === ":memory:") {
      throw new Error("SUGGESTION_REPAIR_REQUIRES_PERSISTED_BACKUP");
    }
    const backupPath = backupDatabase(database.db, options.databasePath, DATABASE_VERSION);
    if (!backupPath) throw new Error("SUGGESTION_REPAIR_BACKUP_FAILED");
    const projection = database.run(() => suggestions.repair(projectId, documentId, true));
    return { backupPath, projection };
  },
}

The checksum includes projection version, coverage, and state:

export function suggestionProjectionChecksum(state: PersistedSuggestionState, coverage: number) {
  return createHash("sha256").update(JSON.stringify({ version: SUGGESTION_PROJECTION_VERSION,
    coverage, state })).digest("hex");
}

Things future maintainers should not gloss over

Checkpoint creation happens after the command transaction. In StorageOperations, createCheckpoint is called after transactions.run. That is safe for data derivation because checkpoints can be recreated, but a checkpoint error could still make a command request fail after the mutation has already committed.

Outbox delivery rebuilds from history for each fact-backed event. OutboxRepository.parseRow creates a new SuggestionRepository and loops through repository.history(...) until the delivered event. It does not use checkpoints in this read path. That is simple and correct, but it is a performance area to watch if suggestion history grows.

The projection invariant is now two-dimensional. A correct projection must have both the expected revision and the expected coveredThroughSequence. The checksum also includes coverage. Repair or import code that updates state without those fields will undermine verification.

Legacy adapter paths still exist. compareAndPut and recordReceipt remain in the port/repository for compatibility with older call shapes. New suggestion mutation paths should prefer appendFacts and recordCommandReceipt.

Comprehension check

Use these to check whether the new data path is loaded in your head before you modify it.

1. A writer pins a suggestion. Which durable rows should exist after the command succeeds?
A command receipt in suggestion_command_receipts, a fact in suggestion_event_history, an advanced suggestion_projection row, and a fact-backed row in event_outbox with suggestion_event_id set and event_json null. The storage integration test checks this shape after publish plus pin.
2. What prevents replay from skipping sequence 3 and applying sequence 4?
applySuggestionFact requires event.sequence === coveredThroughSequence + 1. If the next event is not contiguous, it throws SUGGESTION_EVENT_SEQUENCE_GAP. The domain test covers this with a sequence-2 event applied to coverage 0.
3. Why does the outbox have to rebuild projection state when reading a fact-backed suggestion event?
The durable row stores a reference to a fact, not a full legacy event payload. Existing consumers expect suggestion.event with suggestionRevision and state, so parseRow replays history up to the referenced fact and materializes that payload.
4. What would break if repair updated state_json but left covered_through_sequence unchanged?
Verification would no longer describe the same projection boundary. The checksum includes coverage, rebuild starts from checkpoints/history coverage, and append uses covered_through_sequence to assign the next fact sequence. A mismatched state/coverage pair can cause checksum mismatch or future sequence conflicts.
5. If an older local database exists, where is the migration path into schema 7?
There is none in the active registry. MINIMUM_SUPPORTED_DATABASE_VERSION equals DATABASE_VERSION and DATABASE_MIGRATIONS is empty, so older non-empty files are rejected and preserved during alpha.
