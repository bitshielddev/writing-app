Comprehension review

Desktop code now has enforceable inward dependency seams.

This staged change is mainly an architectural extraction: storage use cases move into application code, small policy functions move into domain code, Pi SDK details move behind an adapter, and a boundary test checks that the direction stays intact.

The path through this diff

The product behavior is meant to stay mostly the same. The important shift is where behavior lives and which modules are allowed to know about runtime details such as SQLite, Node files, Electron, and the Pi SDK.

domain Pure policies: event cursor classification, replay limits, revision checks, and the existing agent loop state machine. New or moved files: desktop/domain/event-sequence.ts, revisions.ts, agent-loop.ts
application Use cases and ports. Storage operations coordinate stores, transactions, files, clock, identity, outbox, and dispatch without importing storage adapters. New files: storage-operations.ts, storage-ports.ts, agent-session-port.ts
infrastructure Runtime adapters. SQLite repositories, file IO, event dispatch, and Pi session conversion implement application-facing ports. Moved or updated files under desktop/storage and desktop/infrastructure/agent
The diff also adds scripts/desktop-boundaries.test.mjs. That makes the layer rule executable instead of relying only on documentation.

What actually happens during document.save

The save path still preserves the important behavior from the old implementation: reject stale revisions before writing the draft file, serialize concurrent saves, write the draft mirror, update SQLite inside a transaction, enqueue a durable event, and dispatch the outbox after the transaction succeeds.

private async performDocumentSave(input: Params<"document.save">): Promise<DocumentSnapshot> {
  if (input.documentId !== this.deps.documentId) throw new Error("Invalid document identity");

  const current = this.deps.documents.get(this.deps.documentId);
  assertDocumentRevision(input.expectedRevision, current.revision);
  if (
    JSON.stringify(input.blocks) === JSON.stringify(current.blocks) &&
    input.markdown === current.markdown
  ) return current;

  await this.deps.files.writeDraft(input.markdown);
  let saved: DocumentSnapshot;
  try {
    saved = this.deps.transactions.run(() => {
      assertDocumentRevision(
        input.expectedRevision,
        this.deps.documents.get(this.deps.documentId).revision,
      );
      const now = this.deps.clock.now();
      const document = this.deps.documents.save(
        this.deps.documentId,
        input.blocks,
        input.markdown,
        now,
      );
      this.deps.projects.incrementRevision(this.deps.projectId, now);
      this.deps.outbox.enqueue({
        type: "document.saved",
        document,
        projectRevision: this.deps.projects.get(this.deps.projectId).revision,
      });
      return document;
    });
  } catch (error) {
    try {
      await this.deps.files.repairDraft(
        this.deps.documents.get(this.deps.documentId).markdown,
      );
    } catch (repairError) {
      throw new AggregateError(
        [error, repairError],
        "Document save failed and draft repair failed",
        { cause: repairError },
      );
    }
    throw error;
  }
  await this.deps.dispatcher.dispatch();
  return saved;
}
The double revision check is intentional in the changed code: the first check protects the draft mirror from a stale write, and the second check protects the database transaction from a race after the file write starts.

Validation moved to the RPC edge

StorageOperations now accepts typed operation parameters. Runtime validation still exists, but it is concentrated in createStorageRequestHandler. The handler parses the incoming params before it calls the operation and parses the result before returning it.

return async (method: string, params?: unknown) => {
  if (!Object.prototype.hasOwnProperty.call(operationMap, method)) {
    throw new Error("Unknown storage operation");
  }
  const name = method as StorageRpcMethod;
  const input = parseOrContractError(
    StorageOperationContracts[name].params,
    params,
    `storage.${name}.params`,
  );
  const result = await (operationMap[name] as Operation)(input);
  return parseOrContractError(
    StorageOperationContracts[name].result,
    result,
    `storage.${name}.result`,
  );
};
Maintenance risk: if future code calls StorageOperations directly outside tests or a typed application layer, it will bypass runtime schema validation. That is not a new product feature, but it is an interface change caused by moving validation to the RPC edge.

Small policies now have names

The domain layer gets small functions that were previously embedded inside infrastructure or process code. That makes them easier to test and gives future code a shared vocabulary for the same rules.

PolicyCurrent callerBehavior grounded in code
classifyEventSequence desktop/agent.ts Incoming sequence <= observed is a duplicate; exactly observed + 1 is next; anything larger is a gap.
clampReplayLimit OutboxRepository.replay Replay limits are clamped between 1 and 100.
nextAcknowledgedSequence OutboxRepository.acknowledge Ack past stream head throws; otherwise acknowledgements are monotonic.
assertDocumentRevision and assertSuggestionDocumentRevision StorageOperations Expected revisions must be integers and equal the current revision, with different error strings for document save and suggestion mutation paths. 
export function classifyEventSequence(
  observedSequence: number,
  incomingSequence: number,
): EventCursorDecision {
  if (incomingSequence <= observedSequence) return "duplicate";
  if (incomingSequence === observedSequence + 1) return "next";
  return "gap";
}

export function nextAcknowledgedSequence(current: number, requested: number, head: number) {
  if (requested > head) throw new Error("ACKNOWLEDGEMENT_PAST_STREAM_HEAD");
  return Math.max(current, requested);
}

The Pi session is no longer the application interface

desktop/agent.ts previously held a Pi AgentSession directly. It now depends on AgentSessionPort, while PiAgentSessionAdapter owns the Pi-specific mapping.

export interface AgentSessionPort {
  readonly id: string;
  readonly isBusy: boolean;
  prompt(instruction: string): Promise<void>;
  abort(): Promise<void>;
  dispose(): void;
  subscribeActivity(listener: (activity: AgentActivityInput) => void): void;
}
export class PiAgentSessionAdapter implements AgentSessionPort {
  constructor(private readonly session: AgentSession) {}

  get id() {
    return this.session.sessionId;
  }

  get isBusy() {
    return this.session.isStreaming;
  }

  subscribeActivity(listener: (activity: AgentActivityInput) => void) {
    this.session.subscribe((event) => {
      activitiesFromSessionEvent(event).forEach(listener);
    });
  }
}

In desktop/agent.ts, that changes reads from session.sessionId to session.id and session.isStreaming to session.isBusy. The production behavior is preserved by the adapter mapping.

The guardrail that keeps this refactor from drifting

The new boundary test scans non-test TypeScript files under desktop/domain and desktop/application. It rejects direct imports of Electron, Node built-ins, Pi, storage adapters, and infrastructure adapters according to the layer being checked.

const forbiddenRuntimeImports = /^(electron|node:|@earendil-works\/pi-coding-agent)/;

export function prohibitedImports(layer, source) {
  return importSpecifiers(source).filter((specifier) => {
    if (forbiddenRuntimeImports.test(specifier)) return true;
    if (layer === "domain") {
      return specifier.includes("/application/") ||
        specifier.includes("/infrastructure/") ||
        specifier.includes("/storage/");
    }
    return specifier.includes("/infrastructure/") || specifier.includes("/storage/");
  });
}

The test does not parse TypeScript ASTs; it uses a regex over import statements. That is enough for the imports shown in the codebase, but dynamic imports or unusual formatting could escape it. The evidence for this limitation is the regex implementation itself.

Why this change fits the recent history

29b9eef and 5372f2f started splitting oversized modules and layering storage.

2a290da introduced contract-validated process boundaries, which explains why the current diff keeps parsing at the RPC edge.

2f264db, cbcb746, and b3763b9 made storage, durable events, replay, acknowledgement, and compatibility more central. The current refactor pulls the policy pieces of those areas inward so they are not owned by SQLite or process modules.

0f6bef4 tightened acknowledgement/subscription behavior. The new cursor classifier in desktop/domain/event-sequence.ts follows that same line of work by naming duplicate, next, and gap decisions.

That history supports the interpretation that this is an architecture hardening change rather than a new user-visible feature. The commit messages alone do not prove product intent, so this review treats that as an inference from the log plus the current diff.

What the tests now protect

  • desktop/domain/policies.test.ts tests event cursor, replay limit, acknowledgement, and revision policies without SQLite, files, Electron, or Pi.
  • desktop/infrastructure/agent/pi-agent-session.test.ts verifies that the adapter satisfies AgentSessionPort and forwards prompt/abort/dispose/subscribe to the Pi session.

Things future maintainers should keep in mind

  • Keep runtime validation at the RPC edge if application operations continue to accept typed params. Moving direct calls into production code should come with either typed construction or explicit validation.
  • When adding a new storage operation, update both the shared contract and the RPC map. The map now casts validated params into operation-specific types, so a missing entry is the place a new method would fail.
  • If a domain helper starts needing Date.now(), UUIDs, SQLite, file IO, or Pi types, that behavior probably belongs in an application port or infrastructure adapter instead.
  • The import boundary test is a guardrail, not a full architecture checker. It catches normal static imports; it does not understand every TypeScript syntax shape.

Pause and check your mental model

1. A stale document.save request arrives. Which changed code path prevents the draft mirror from being overwritten?

performDocumentSave calls assertDocumentRevision(input.expectedRevision, current.revision) before files.writeDraft(input.markdown). That first check is the pre-file-write guard.

2. Why can StorageOperations now be tested without SQLite or Node file IO?

It depends on interfaces from storage-ports.ts, such as DocumentStore, WorkspaceFiles, TransactionManager, Clock, and IdentityGenerator. Tests can provide fake implementations of those ports.

3. If an agent process sees sequence 6 after observing sequence 3 on the same stream, what happens?

classifyEventSequence(3, 6) returns "gap". In desktop/agent.ts, that causes the agent to call agent.seed, reset its observed stream cursor from the seed, and emit the seeded revision.

4. What would break architecturally if desktop/application/storage-operations.ts imported node:crypto again?

The boundary test would report a prohibited runtime import for an application module. More importantly, the application use case would again own a runtime decision that is now supplied by the composition root through IdentityGenerator.

5. Where would you start if Pi activity stopped appearing in the runtime panel after this refactor?

Start at PiAgentSessionAdapter.subscribeActivity. It subscribes to the Pi session, converts each session event through activitiesFromSessionEvent, and forwards each activity to the listener passed by desktop/agent.ts.
